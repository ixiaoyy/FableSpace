# 玩家扮演 NPC 角色扮演模式 - 头脑风暴

## 背景

当前 FableMap 的玩法是：
- 玩家作为**访客**进入酒馆
- 与 **AI NPC** 对话
- AI NPC 由**店主**配置

## 新想法

**玩家自己扮演 NPC**，打破纯 AI 对话模式。

### 核心概念

1. **角色扮演模式** - 玩家可以有特定的剧情、身份和任务
2. **抢占机制** - 可以"顶替"某个 AI NPC 或新增一个角色槽位
3. **常驻身份** - 玩家可以长期扮演某个角色，成为这个酒馆的"真人 NPC"
4. **真人社交** - 其他访客进来看到的是真人在扮演，而不是 AI

### 场景示例

```
场景：夜莺酒馆

原状态：
- 酒馆内有 3 个 AI NPC：老板、酒保、吟游诗人
- 访客进来和 AI 对话

新玩法：
- 玩家 A 抢占"吟游诗人"角色，开始扮演
- 玩家 B 进入酒馆，看到"吟游诗人"是玩家 A 在扮演
- 玩家 B 可以和玩家 A 角色扮演互动
- 玩家 A 离线后，"吟游诗人"恢复为 AI 代理
```

## 核心问题

### Q1: 酒馆模式

酒馆需要区分两种模式吗？
- **体验模式**（当前）：纯 AI NPC，访客进来和 AI 聊天
- **扮演模式**：允许玩家扮演角色
- **混合模式**：AI + 玩家角色共存

### Q2: 角色所有权

- 店主创建的 AI NPC可以被抢占吗？
- 玩家扮演的角色是临时的还是永久的？
- 玩家扮演时，店主还能控制这个角色的行为吗？

### Q3: 交互方式

- 玩家扮演时，是实时聊天还是异步消息？
- 多个玩家同时扮演时如何协调？
- 玩家不在线时，AI 如何代理？

### Q4: 激励系统

为什么玩家愿意扮演 NPC？
- 社交认可/声望
- 任务奖励
- 角色成长
- 付费特权？

## 2026-04-24 收束结论

当前结论：**暂不开发，只做未来方向留档**。

原因：

- 当前 `docs/WHAT_NOT_TO_BUILD.md` 明确不做访客间社交、访客直接对话、访客间消息传递。
- 玩家扮演 NPC 本质上会引入真人与真人互动、在线状态、消息路由和审核机制，超出当前“访客与 AI NPC 互动”的主链路。
- 如果未来要重启，需要先由用户明确调整产品边界，再重新开 Trellis task 做需求评审、数据模型设计和安全策略。

本 task 不进入实现，不新增 API / schema / WebSocket / 前端状态。
