# 头脑风暴：地点类型 + Home 概念 + 任务系统

## 背景

当前产品只支持"酒馆"一种地点类型。新的方向是：
- 扩展地点类型：酒馆、咖啡店、奶茶店、牛排店、服装店、学校等现实生活中的店铺 / 场所
- 增加"Home"概念：普通用户也可以有自己的主页
- Home 可以设置角色、形象，能被其他用户拜访
- 加入任务/玩法系统


## 2026-04-27 用户补充：整体风格大改方向

用户明确补充：当前“酒馆”不应被理解为唯一形态，而只是“现实中某种店 / 场所”的一种表现。FableMap 后续整体风格希望从“全都是酒馆”扩展为“真实地图上的生活地点 + 个人 Home 网络”。

### 核心产品语义变化

- **酒馆只是 Place 的一个类型**：它代表“店 / 场所”的一种氛围，不是平台唯一核心名词。
- **Place 应覆盖现实生活中存在的店铺 / 场所类型**：咖啡店、奶茶店、牛排店、学校等都可以作为地图上的可拜访地点。
- **整体视觉与文案需要大改**：不再让用户感觉所有入口、卡片、详情页、默认体验都围绕“酒馆”；应逐步改为“地点 / 家 / 生活场景 / 角色关系”的综合体验。
- **地图仍是现实锚点**：这些地点类型应优先对应现实生活中存在的店铺或场所概念，而不是无锚点虚拟空间。

### Home 概念深化

用户希望不只是“店长开店”的形式，普通人也应该拥有自己的 Home：

- 普通用户可以拥有 Home / 家。
- Home 可以设置整体形象，例如封面、主题、空间氛围。
- Home 内的角色 / 家庭成员也可以设置形象。
- Home 可以被其他用户拜访。
- 用户之间可以互相拜访 Home，但需要后续设计访问边界，避免演变成无边界社交网络。

### 家庭成员与跨地点关系

Home 不只是静态主页，还可以承载“家庭成员”关系：

- Home 主人可以设置家庭成员。
- 可以添加“孩子”等家庭角色。
- 家庭成员不限种族、不限身份，可以是人类、动物、幻想种族、机器人、物件等。
- 甚至可以把一块石头当作孩子。
- 家庭成员可以与其他地点产生关系：例如孩子被送去指定学校后，对应学校里会真的多一个学生。
- 这意味着未来可能需要“跨 Place/Home 的角色关系同步”：Home 中的成员可以出现在学校、店铺或其他地点的成员 / 访客 / 常驻角色列表中。

### 非生物家庭成员交互规则

用户特别说明：如果家庭成员是非生物类，例如“石头孩子”，它不应该像普通角色一样讲话和互动。

候选规则：

- 非生物 / 不具备表达能力的成员可以作为家庭成员和视觉对象存在。
- 它可以被展示、被命名、被记录关系，但默认不主动说话。
- 当访客尝试与其对话时，系统可以返回沉默反馈，例如：`...`
- 这类成员不应被强行补全成人格化 NPC，避免平台越权创作。

### 新方向的产品关键词

- 地点类型平台，而不是单一酒馆平台。
- Home / 家，而不是只有店铺经营。
- 角色关系网络，而不是孤立 NPC。
- 可拜访生活场景，而不是传统地图 App。
- 受控互访，而不是开放社交网络。
- 结构化关系同步，而不是 AI 自由生成世界。

## 核心概念转变

### 从单一到多元
| 当前 | 新方向 |
|------|--------|
| 酒馆 (Tavern) | 地点 (Place) |
| 店主 (Owner) | 主人 (Host) |
| 访客 (Visitor) | 访客/探索者 (Visitor/Explorer) |
| - | 普通用户 Home |
| - | 任务 (Quest/Gameplay) |

### 用户角色扩展
| 角色 | 描述 | 能力 |
|------|------|------|
| 探索者 (Explorer) | 浏览地图、发现地点、拜访 | 探索、对话、完成任务 |
| 主人 (Host) | 创建和运营地点 | 创建地点、管理 NPC、设置营业时间 |
| 普通用户 (User) | 拥有自己的 Home | 设置 Home 形象、邀请朋友拜访 |
| Home 主人 (Homeowner) | 拥有 Home | 自定义 Home、设置角色、被拜访 |

## 产品方向候选

### 方向 A：地点类型平台
- 地图上显示多种类型的地点
- 每种地点类型有不同的默认模板和氛围
- 主人可以选择地点类型（酒馆/咖啡店/奶茶店/学校等）
- 统一的拜访和对话机制

### 方向 B：Home + 地点混合
- 所有用户都有 Home
- 用户可以选择是否"开门营业"成为公开地点
- Home 和公开地点使用同一套数据模型
- 探索者可以探索公开地点，也可以拜访朋友的 Home

### 方向 C：Home 优先
- 产品核心转向"个人主页"
- 地点发现是次要功能
- 用户主要在自己的 Home 互动
- 公开地点作为"热门景点"推荐

## Home 概念设计

### Home 数据模型（候选）

```typescript
interface Home {
  id: string;
  owner_id: string;
  name: string;                    // Home 名称
  description: string;
  avatar?: string;                 // Home 头像
  theme: 'cozy' | 'modern' | 'vintage' | 'futuristic';  // Home 风格
  characters: Character[];         // Home 内的角色
  visit_settings: {
    public: boolean;              // 是否公开
    approval_required: boolean;    // 是否需要审批
    friends_only: boolean;        // 仅好友可访
  };
  // 时间系统（复用酒馆时间系统）
  timezone?: string;
  operating_hours?: OperatingHours;
}
```

### Home vs Place 对比

| 维度 | Place (酒馆等) | Home |
|------|----------------|------|
| 创建门槛 | 较高（需要经营） | 低（人人都有） |
| 公开性 | 通常公开 | 可私有 |
| 访问控制 | 密码/审批 | 朋友/审批 |
| 运营目标 | 服务探索者 | 社交/展示 |
| 典型场景 | 深夜食堂、便利店 | 朋友聚会、个人展示 |

## 任务/玩法系统

### Quest 类型候选

| 类型 | 描述 | 示例 |
|------|------|------|
| 地点探索 | 探索特定地点 | "找到并拜访 5 家咖啡店" |
| NPC 对话 | 与特定 NPC 互动 | "和老王聊一次天" |
| Home 拜访 | 拜访朋友 Home | "拜访 3 个朋友的 Home" |
| 社交互动 | 与其他用户互动 | "给 5 个地点留下好评" |
| 创作任务 | 创建内容 | "创建一个新地点" |

### Quest 数据模型（候选）

```typescript
interface Quest {
  id: string;
  title: string;
  description: string;
  type: 'exploration' | 'social' | 'creation' | 'collection';
  objectives: QuestObjective[];
  rewards: QuestReward[];
  visibility: 'public' | 'private';
}

interface QuestProgress {
  quest_id: string;
  user_id: string;
  status: 'active' | 'completed' | 'abandoned';
  progress: number;           // 0-100
  completed_objectives: string[];
  started_at: string;
  completed_at?: string;
}
```

## 待讨论问题

1. **Home 是否必须有地点坐标？**
   - 选项 A：Home 基于真实地理位置
   - 选项 B：Home 是纯线上虚拟空间

2. **Place 和 Home 是否共用同一数据模型？**
   - 选项 A：是，共用 `Place` 模型，通过 `is_home` 字段区分
   - 选项 B：否，分开模型

3. **地点类型的数量是否有限？**
   - 选项 A：固定几种类型，模板化
   - 选项 B：用户可自定义类型

4. **Home 拜访的隐私边界？**
   - 是否需要双向确认？
   - 是否记录拜访历史？
   - 是否支持黑名单？

5. **任务系统由谁创建？**
   - 选项 A：平台预设任务
   - 选项 B：用户可创建任务
   - 选项 C：AI 自动生成任务


6. **Home 家庭成员是否是核心 MVP？**
   - 选项 A：MVP 只做 Home 形象和角色展示，家庭成员关系后置
   - 选项 B：MVP 支持家庭成员，但不做跨地点同步
   - 选项 C：MVP 即支持“孩子送去学校后学校多一个学生”的跨地点关系

7. **非生物家庭成员的交互边界？**
   - 选项 A：作为静态成员，只展示不对话
   - 选项 B：允许被问候，但统一返回 `...`
   - 选项 C：允许主人手动赋予可说话设定，否则默认沉默

8. **互相拜访的关系模型？**
   - 选项 A：只通过邀请链接 / 访问码拜访，不做好友系统
   - 选项 B：做 Home 访问白名单，但不做动态墙 / 私信
   - 选项 C：做好友关系（风险：可能触碰“无边界访客社交”负面清单）
