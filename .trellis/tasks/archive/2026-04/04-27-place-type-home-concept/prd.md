# 新方向产品收敛：多地点类型与 Home 概念 PRD

## 背景

`.trellis/tasks/04-27-place-type-home-concept/brainstorm.md` 已记录一次新方向头脑风暴：从单一“酒馆”扩展到多种现实地点类型，并讨论普通用户 Home、角色扩展、拜访关系、家庭成员、跨地点角色关系和任务/玩法系统。

该方向会触碰 FableMap 的核心产品定义和负面清单边界，不能直接进入实现，需要先收敛成明确的产品决策与后续任务拆分。


## 用户最新方向说明

用户明确希望做一次整体风格大改：

- “酒馆”只是现实店铺 / 场所的一种形式，不应继续成为所有体验的唯一外壳。
- 平台应能承载咖啡店、奶茶店、牛排店、学校等现实生活中存在的地点类型。
- 不只是店长开店；普通用户也应有 Home / 家的概念。
- 用户可以给自己的 Home 设置形象，给角色 / 家庭成员设置形象。
- 用户之间可以互相拜访 Home。
- Home 可以设置家庭成员，例如孩子。
- 家庭成员可以被送去指定学校；对应学校应真实增加一个学生，形成跨 Home / Place 的结构化关系。
- 家庭成员不限种族、不限身份；甚至可以把石头当作孩子。
- 非生物类家庭成员默认不讲话、不交互；被问到时可以表现为 `...`，而不是被平台强行人格化。

## 初步收敛判断

这个方向不只是“新增地点类型”，而是把产品从 **Tavern-only** 推向 **Place + Home + Relationship Graph**：

- **Place**：真实地图上的店铺 / 场所，Tavern 是其中一种。
- **Home**：普通用户的个人空间 / 家，可被受控拜访。
- **Member / Character**：可以属于 Home，也可能与学校、店铺等 Place 建立关系。
- **Relationship**：记录“某成员属于某个 Home”“某孩子就读某学校”等结构化关系。

必须重点评审的边界：

- 真实地图锚点不能丢。
- 互访不能滑向无边界好友 / 私信 / 动态墙社交。
- 学校与孩子相关能力需要隐私、安全和内容边界。
- 平台不能绕过主人确认自动生成家庭成员人格或故事。

## 目标

把头脑风暴中的发散内容收敛为一份可评审的产品方向决策：

- 明确 FableMap 是否从 Tavern-only 扩展为 Place-first。
- 明确 Home 是否存在，以及它是否必须绑定真实坐标。
- 明确 Explorer / Host / 普通用户 / Home 主人的边界。
- 明确地点类型策略：固定模板、有限枚举，还是可扩展类型。
- 明确 Home / 家庭成员 / 跨地点关系是否进入 MVP。
- 明确非生物家庭成员的展示与交互规则。
- 明确任务/玩法系统是否符合“赛博酒馆 UGC 平台”定位，避免走向传统 RPG 或通用社交。

## 关键待决策问题

1. Home 是否必须绑定真实地理坐标？
2. Place 与 Home 是否共用模型，还是保留 Tavern 作为核心模型？
3. 地点类型是固定有限类型，还是允许用户自定义？
4. Home 拜访是否会引入被禁止的无边界访客社交？
5. 任务/玩法系统是否只是酒馆内引导与回访机制，还是会滑向传统 RPG 任务？
6. Home 家庭成员是否进入 MVP？如果进入，是否允许跨地点同步到学校等 Place？
7. 非生物家庭成员是否默认沉默，还是允许主人手动赋予人格？
8. 互访采用邀请/白名单，还是需要更强的关系系统？

## 非目标

- 本任务不直接修改 Tavern / TavernCharacter / WorldInfoEntry 等 Schema。
- 本任务不直接实现 Home、Place 类型或任务系统。
- 本任务不引入访客好友、私信、动态墙；Home 互访只能作为受控访问机制讨论。
- 本任务不直接实现跨地点角色同步，只评估“孩子进入学校后学校多一个学生”等关系模型。
- 本任务不允许脱离真实地图锚点的自由空间。
- 本任务不设计战斗、等级、装备、传统 RPG 任务。

## 验收标准

- [ ] 产出一个明确的产品方向结论：继续 Tavern-only、Place-first、Home-first，或分阶段组合方案。
- [ ] 每个结论都显式对照 `docs/WHAT_NOT_TO_BUILD.md`，说明允许/禁止边界。
- [ ] 如果保留 Home，明确 Home 的坐标、隐私、访问、内容责任边界。
- [ ] 如果保留 Home 家庭成员，明确成员类型、形象、交互能力和非生物沉默规则。
- [ ] 如果支持家庭成员跨地点关系，明确学校/Place 的成员同步规则和隐私边界。
- [ ] 如果保留多地点类型，明确类型枚举、命名、与 Tavern 的关系。
- [ ] 如果保留任务/玩法系统，明确它是“酒馆体验引导/回访反馈”，不是传统 RPG 系统。
- [ ] 拆分出后续可实现的 Trellis 子任务或明确暂缓。

## 建议下一步

1. 先做产品边界评审，不动代码。
2. 对照权威文档和负面清单标出冲突点。
3. 收敛出 1 个主方向和最多 2 个后续实施任务。
4. 优先决定 MVP 是“多地点类型”先行，还是“Home + 家庭成员”先行，避免一次性重写全部产品。
