# 多地点类型与 Home 概念产品收敛决策

> 日期：2026-04-27
> 任务：`04-27-place-type-home-concept`
> 状态：AI 草案，供用户评审；本轮不修改代码、不修改持久 Schema、不改全局主线文档。
> 依据：`README.md`、`docs/PRODUCT_BRIEF.md`、`docs/FABLEMAP_TAVERN_PLATFORM.md`、`docs/WHAT_NOT_TO_BUILD.md`、`docs/WORLD_SCHEMA.md`。

## 1. 总结论

建议采用 **Place-first, Tavern-compatible（地点优先，酒馆兼容）** 的分阶段方向。

含义：

1. **产品语义可以从 Tavern-only 收敛为 Place-first**：酒馆是第一种、已落地的 Place 类型，不再要求所有真实生活场景都被文案包装成“酒馆”。
2. **实现上短期继续保留 Tavern 作为唯一稳定持久实体**：当前 `Tavern` 已承载坐标、访问控制、角色、世界书、玩法、记忆、时间系统等主链路；本任务不直接重命名模型或新增 Schema。
3. **Home 可以进入产品路线，但不进入当前 MVP 实现**：Home 是“普通用户拥有的受控私密地点/个人空间”，必须保持真实地图锚点和主人主权，默认不公开发现。
4. **家庭成员与跨地点关系图先作为后续设计，不进入第一阶段实现**：尤其是“孩子送去学校，学校多一个学生”会触碰隐私、权限、跨主人写入和未成年人语义，不能直接做成自动同步。
5. **任务/玩法继续限定为地点内的轻量体验引导**：沿用当前 `GameplayDefinition` 方向，不做传统 RPG 任务、等级、装备、排行榜或平台级全局任务系统。

推荐路线不是 “Home-first” 或一次性重写为 “Place + Relationship Graph”，而是：

```text
Phase A：术语与入口收敛（Tavern 仍为实现模型）
Phase B：有限 Place 类型 MVP（真实坐标 + 模板/筛选/文案）
Phase C：Home MVP（私密锚点 + 受控拜访 + 形象/成员展示）
Phase D：关系图/学校同步专项设计（审批、隐私、非生物规则）
```

## 2. 对关键问题的决策

| 问题 | 决策 | 理由 / 边界 |
|------|------|-------------|
| Home 是否必须绑定真实地理坐标？ | **是，内部必须有真实坐标；公开展示可做隐私保护。** | `WHAT_NOT_TO_BUILD.md` 禁止无锚点自由空间。Home 可以粗略显示、隐藏精确地址或仅邀请可见，但不能成为脱离地图的自由房间。 |
| Place 与 Home 是否共用模型？ | **概念上同属 Anchored Space；实现上先不共用新模型。** | 当前 `Tavern` 是稳定 Schema。后续可先做兼容层 / 命名层，再决定是否引入 `place_type` 或新 `Place` 模型。 |
| 地点类型固定还是自定义？ | **MVP 用有限枚举，不开放任意自定义。** | 避免术语扩散和 UI/模板不可控。建议首批：`tavern`、`cafe`、`milk-tea-shop`、`restaurant`、`convenience-store`、`bookstore`、`school`、`home`（Home 独立受控）。 |
| Home 拜访是否引入社交？ | **只做受控访问，不做好友系统。** | 允许邀请链接、访问码、白名单/审批等；禁止动态墙、私信、陌生人匹配、全局好友关系。 |
| 任务/玩法系统是否保留？ | **保留地点内轻量玩法，不做全局 RPG。** | 继续对齐 `GameplayDefinition`：店主定义、单地点内运行、可回放 fallback；不做战斗/等级/装备/排行榜。 |
| Home 家庭成员是否进入 MVP？ | **不进入第一阶段 MVP；先做设计。** | 家庭成员涉及隐私、形象、角色权限和 AI 人格边界，尤其“孩子”语义高风险。 |
| 是否允许非生物家庭成员？ | **允许作为成员/视觉对象；默认沉默。** | 非生物或无表达能力成员不应被平台强行人格化。被问到时默认返回 `...`；只有主人明确配置为可说话角色时才参与对话。 |
| “孩子送去学校后学校多一个学生”怎么处理？ | **不能自动写入他人学校；必须有目标 Place 主人/规则审批。** | 若学校由同一主人拥有，可作为私有关系试点；跨主人学校只能产生待审批关系，不能直接改公开学生列表。 |

## 3. 允许与禁止边界

### 3.1 允许

- 把“酒馆”调整为 **Place 的一种类型**，用于咖啡店、奶茶店、牛排店、便利店、学校等真实生活地点语义。
- 继续让每个 Place 必须挂接真实坐标；真实地址可按访问权限做展示级别控制。
- 给 Place/Home 设置封面、氛围图、角色/NPC/成员形象。
- 平台提供模板、AI 草稿和创作辅助，但所有可发布内容必须由主人确认。
- Home 通过邀请链接、访问码、私密/审批等方式被拜访。
- 非生物成员作为静态对象、关系对象或沉默成员存在。
- 地点内玩法用当前轻量 `GameplayDefinition` 思路表达，如导览、问答、回访提示、委托板。

### 3.2 禁止

- 不允许无真实坐标的自由 Home / 虚拟空间。
- 不允许平台自动生成并上线 Home 成员、学生名单、NPC 人格或故事。
- 不允许把 Home 互访扩展成好友系统、私信、动态墙、陌生人匹配。
- 不允许把学校做成未成年人真实身份/真实学生数据收集系统。
- 不允许跨主人自动写入学校、店铺或其他 Place 的公开成员列表。
- 不允许全局传统 RPG 任务、战斗、等级、装备、排行榜。
- 不允许以“多地点类型”为理由夹带传统地图 App 的路线规划、POI 评分、商家入驻等功能。

## 4. 与负面清单对照

| `WHAT_NOT_TO_BUILD.md` 禁区 | 本决策如何避免 |
|----------------------------|----------------|
| 无锚点自由空间 | Home 内部仍要求真实坐标；只是展示时可隐私保护。 |
| 无边界访客社交 | Home 互访只走邀请/审批/访问码，不做好友、私信、动态墙。 |
| 平台绕过主人确认发布内容 | Place/Home/成员/NPC/学生关系都必须主人确认；AI 只能出草稿。 |
| 传统地图 App 功能 | 多地点类型只服务可进入的 AI/角色体验，不做导航、评分、商家入驻。 |
| 传统 RPG 任务 | Gameplay 只做地点内轻量互动与回访，不做战斗/等级/装备/排行。 |
| 敏感数据暴露 | Home 坐标、家庭成员、学校关系默认按私密数据处理；公开前需要显式授权。 |

## 5. Home 设计边界草案

### 5.1 Home 的定位

Home 是普通用户拥有的 **受控、可拜访、真实锚点个人空间**，不是开放社交主页。

Home 的主人等价于 Host/Homeowner：

- 能设置 Home 名称、封面、空间氛围。
- 能添加成员/角色/物件。
- 能决定访问方式：私人、邀请链接、访问码、审批、白名单。
- 对 Home 内内容负最终责任。

### 5.2 坐标与隐私

- 内部数据必须有坐标，用于保持 FableMap 的真实地图锚点。
- 访客侧不一定展示精确门牌或坐标：可显示城市/区域级、模糊点位、或仅在受邀后显示。
- 私密 Home 不进入公共发现列表。
- 后续实现需要明确定义 public payload 与 owner payload 的差异。

### 5.3 成员与表达能力

建议把 Home 内对象分为三类：

| 类型 | 默认表现 | 是否能对话 |
|------|----------|------------|
| Conversational Character | 角色卡/NPC，有 `system_prompt` 等 | 可以，必须主人配置或确认 |
| Silent Member | 家庭成员/宠物/非生物对象 | 默认不说话，被问到返回 `...` |
| Display Object | 纯展示摆设/纪念物 | 不对话，只展示 |

关键规则：平台不得因为用户写了“石头孩子”就自动补全成会说话的人格。只有主人明确配置为可说话角色，才进入 AI 对话链路。

## 6. 学校与跨地点关系边界草案

“孩子送去指定学校，学校多一个学生”应被拆成关系图专项，而不是在 Home MVP 直接实现。

建议规则：

1. **同主人场景**：如果 Home 和 School Place 属于同一主人，可先允许私有关系同步。
2. **跨主人场景**：Home 主人发起 enrollment 请求，School 主人/管理员批准后才出现在学校可见成员列表。
3. **公开展示**：默认只展示昵称/化名/虚构身份，不展示真实未成年人信息。
4. **非生物成员**：可被登记为“特殊学生/成员”，但学校方可选择是否公开展示；默认仍不对话。
5. **平台职责**：只存结构化关系与权限，不自动编造入学故事、成绩、社交关系。

## 7. 地点类型策略草案

MVP 不开放任意自定义类型，先做有限枚举和模板。

推荐首批类型：

| 类型 | 用途 | 备注 |
|------|------|------|
| `tavern` | 当前已实现主线 | 保持兼容，默认类型 |
| `cafe` | 咖啡店 | 白天/轻社交场景 |
| `milk-tea-shop` | 奶茶店 | 年轻、轻快场景 |
| `restaurant` | 牛排店/餐馆 | 正餐、约会、庆祝 |
| `convenience-store` | 便利店 | 24h、夜班、社区 |
| `bookstore` | 书店 | 安静、文艺、阅读 |
| `school` | 学校/教育场所 | 高隐私风险，仅限受控模板 |
| `home` | 个人 Home | 默认私密，不进公共发现 |

实现前必须再决定：这些类型是先作为前端模板/标签，还是进入后端 Schema。该决定属于后续实现任务，不在本任务拍板。
