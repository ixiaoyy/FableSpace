# 产品主线重构

## Goal

重新定义 FableSpace 当前唯一的产品主线，并让仓库入口文档、AI 协作约束、对外定位、前台信息架构和仍保留的兼容代码使用同一套优先级，避免任何审查者继续把历史上的“真实坐标 / 地图 LBS / 店主 UGC / SillyTavern 内容供给”总结成当前产品本质。

当前确认的一句话定位：

> FableSpace 是一个角色故事平台。玩家因想见一个角色而进入其所属的完整故事世界，通过对话和选择建立关系、经历事件，并在回访中延续彼此的故事。

互动历史故事是角色故事平台中的一种玩法，不是平台唯一题材。历史玩法通过原创人物带玩家进入真实历史现场；玩家能改变人物的私有命运，但不能改写固定史实。皇宫、志怪、校园等原创或架空世界继续属于合法产品内容。

## Background

- 2026-07-16 的角色优先改造已经把主链路改为“游玩身份 -> 角色发现 -> 完整故事 Space -> 对话 / 选择 -> 私有连续性”，但这仍可能只是当前产品继续演进中的过渡状态。
- 当前 `README.md` 首屏仍用“真实坐标 + AI NPC + 角色卡 + 连续故事”介绍项目；`AGENTS.md`、`docs/PRODUCT_BRIEF.md`、`docs/FABLESPACE_SPACE_PLATFORM.md` 和 `docs/WHAT_NOT_TO_BUILD.md` 仍把真实坐标与 SillyTavern 兼容写成核心或硬约束。
- 当前代码仍保留地图配置、店主后台、Space 创建、角色卡导入导出、模板、私有 LLM 配置和 Token 统计等大量历史兼容面。这些实现的关键词和文件体量足以压过较新的角色故事入口。
- 现有构建产物证明外部 AI 使用了当前代码和构建数字，却仍把旧兼容层升格为产品战略；因此问题不是简单缓存，而是仓库继续表达了多套相互竞争的产品心智。
- 用户已明确确认“现在已经不是这样的了”，本任务不能把旧的坐标 / UGC / 角色卡平台叙事继续当作当前定位。
- 用户随后明确否决“互动历史故事产品”作为平台总定位；平台本体仍是角色故事平台，互动历史只是一种玩法。
- 项目仍处于开发阶段，用户明确要求大刀阔斧删减，不要求兼容旧产品概念、旧路由、旧字段或历史开发数据。重构以清晰的新产品为优先，不保留新旧双轨。

## Requirements

- 当前 P0 必须服务角色故事平台定位，不再把坐标平台、Space UGC、角色卡生态、店主创作工具或某一种题材玩法作为并列产品本体。
- 通用角色故事合同是：角色属于一个有稳定正史、规则、当前事件和后果的完整故事世界；玩家通过对话和选择建立私有关系与连续性；不同角色必须从自身欲望、秘密、语言和处境形成不同反应。
- P0 每个故事世界只提供一个完整且固定的专属 `PlayerRole`，包含称谓、性别、背景与入场理由；故事稳定后再评估增加其它身份。平台删除全局身份目录和 `male` / `female` 自声明步骤，不再把 `beggar` / “乞丐”外溢为所有世界的固定身份。
- `PlayerRole` 必须符合所属世界的时代、制度、入场理由、已发布正史和角色反应边界。前台只显示短且真实可识别的身份名，具体职责、已知信息和剧情钩子属于后台故事合同，不拼进称谓。
- 已完成的宽街故事保持原有核心：约十岁的原创角色安妮抱着陶罐向玩家讨水，玩家决定给水、追问水源、另找水或求助成年人；`beggar` / “乞丐”改为该故事专属的固定预设身份，不再外溢为所有故事的通用身份。
- 宫廷故事保留“皇帝昏迷、掌印太监封锁寝殿、公主试图闯宫”的核心冲突，固定玩家身份为“小太监”。其传话、跑腿和熟悉宫规的能力必须来自故事背景与具体职责；身份仍没有开门或替任何一方决策的权力，帮助任何一方都应产生可见风险。
- P0 首发内容收敛为两个故事世界：安妮的 1854 年宽街故事用于验证历史玩法，雪夜封宫故事用于验证原创架空玩法。青槐驿与临川大学故事不删除，但移出 P0 公开首发范围；在各自专属身份、角色反应和连续性重新设计完成前不得作为已完成主线展示。
- P0 唯一玩家旅程为：`首页直接发现角色 -> 选择想见的角色 -> 查看一句故事处境与该故事固定身份 -> 登录或确认已有会话 -> 进入开场对话 / 选择 -> 写回关系与记忆 -> 回访时从结果继续`。不再保留全局身份选择页，也不要求玩家先浏览故事世界列表；故事世界作为角色所属背景与连续性边界呈现。
- 进入安妮时自动使用该故事的“乞丐”身份并进入讨水场景；进入魏观海或萧明珠时自动使用雪夜封宫的“小太监”身份，并从所选角色的立场进入同一宫廷事件。
- 当前阶段只提供系统策划、人工审核并可版本化发布的故事内容。普通用户不能创建、编辑或发布故事世界；旧店主 / 创作者入口、创建向导、管理页和发布流程退出产品与代码，不作为隐藏兼容能力保留。未来若重新引入创作者生态，必须在角色故事核心体验稳定后另行设计内容审核、权限和版本合同。
- SillyTavern 角色卡导入、导出及兼容合同整体退出产品与代码，不再作为对外能力、Schema 硬约束或内部架构目标。系统故事仍可保留名称、人设、开场白和示例对白等运行所需内容，但字段与格式只服务 FableSpace 自身的故事、角色和连续性合同。
- 私有连续性是 P0 核心能力：`PlayerStoryState` 按 `玩家 + 故事世界` 隔离，内部再按角色保存关系；系统持久化关键选择、故事进度、关系变化和经过筛选的私有记忆，并在回访时从已有结果继续。状态不跨故事世界传播，不进入公开关系、访客社交或排行榜。
- 首页、角色卡、故事处境和固定身份允许未登录浏览；开始、恢复、推进或重新开始故事必须先登录。所有私有连续性直接归属登录账号，不创建匿名玩家标识，不提供匿名进度绑定、迁移或合并；登录成功后应安全返回玩家原先选择的角色入口。
- 用户界面统一使用“故事世界”；代码实体、API 与 Schema 统一使用 `StoryWorld` / `story_world_id`，路由使用 `/story-worlds/...`。现有 `Space` / `space_id` / `/spaces` 合同整体删除，不保留兼容类型、字段或路由别名；不得退化为含义过宽且易与世界知识混淆的 `World`。
- 故事人物在用户界面统一称“角色”，代码实体与标识统一使用 `Character` / `character_id`，删除 `SpaceCharacter` 命名。`NPC` 只描述角色由 AI 演绎的运行方式，不作为持久化领域实体名。
- 故事预设玩家身份在用户界面显示为“你的身份”，代码实体与标识统一使用 `PlayerRole` / `player_role_id`；每个 `PlayerRole` 必须属于一个 `StoryWorld`。删除 `play_identity` / `play_identity_id`、全局身份目录和用户可编辑身份 Prompt。
- 玩家在一个故事世界中的私有长期状态统一为 `PlayerStoryState`，按 `player_id + story_world_id` 唯一定位；`player_id` 只能由服务端从有效登录会话解析。它包含当前 `PlayerRole`、关键选择、章节 / 节点进度、各 `Character` 关系、私有记忆和回访时间，取代含义宽泛的 `VisitorState`。
- LLM 模型、API Key、服务地址和生成参数统一由系统 / 部署环境管理。删除每个 owner 或故事世界的私有 LLM 配置、管理页面、写入 API 与面向店主的 Token 统计；后端仅保留不泄露密钥的必要调用日志和运维指标，不形成计费产品。
- 系统故事删除 `owner_id`、`public` / `password` / `private`、故事密码和店铺式营业状态，只保留内部发布生命周期 `draft` / `published` / `archived`。公开 API 只返回 `published`；`archived` 停止新玩家进入但保留已有进度与回滚依据。玩家私有数据继续由 `PlayerStoryState` 隔离，不依赖故事访问密码。
- 故事采用“人工剧情骨架 + AI 角色演绎”的混合推进：系统预先定义章节、关键选择、状态变更和允许结局；玩家可点击选择或自由输入，AI 负责按角色语言回应、追问和呈现线索。只有已定义的剧情动作可以推进章节或写入关键状态，运行时 AI 不得创造新正史、跳过条件或自行生成永久结局。
- 角色关系采用“连续好感度 + 故事状态 / 关键标记”的双层模型。自然对话可以依据角色人设与当前语境小幅改变好感度，并让同一角色呈现不同语气、信任和信息开放程度；已审核的关键选择可以造成较大关系变化并开启或关闭分支。剧情不得被少数固定选项完全锁死，也不得只靠重复闲聊无成本刷出重大关系或结局。
- 好感度以内部连续数值存储和计算，不直接向玩家显示。前台只展示角色可感知的关系阶段、态度变化及原因，例如“仍有戒心”“开始信任你”“因隐瞒真相而失望”；不得把角色关系呈现为可刷取的分数面板。
- 故事进行中不得撤销关键选择；到达结局后可以明确“重新开始”，创建新的故事轮次。上一轮只保留结局摘要和关键关系结果，不继续影响新轮次；P0 不提供存档槽、章节回退或多条并行时间线。
- 互动历史是内容玩法层的附加合同：从原创人物的具体困境和请求进入；史实通过地点、物件、证词、对话、选择和结果逐步解锁；经核验的真实时间线、地点、真实人物同场关系、制度阶段和公开结果不可改写。
- 皇宫、志怪、校园等原创或架空世界继续合法，不受历史玩法的固定史实合同约束，但仍必须遵守自身已发布正史与连续性边界。
- 真实坐标、地图和 LBS 不再属于产品合同。故事世界无需绑定现实地点；历史题材记录真实事件地点时使用内容事实与来源表达，不构成通用坐标字段或地图入口。
- 删除 `lat` / `lon` 强制约束、地图配置、地图入口、坐标发现、坐标创建步骤和只服务旧定位的地图代码；不保留新旧双轨。
- 允许重命名或重建开发期 Schema、API、路由、种子与存储，不为旧路由、旧字段、旧开发数据或旧自托管部署保留兼容适配器。
- 权威入口必须只表达一套可排序的产品层级：当前主线、必要底层能力、待重写内容、历史残留、明确不做。
- 不得仅在旧文档后追加新段落；所有会影响 AI 和开发者判断的开头摘要、术语、优先级和交叉引用都必须同步改写。
- 代码和路由必须按最终产品决定分类：当前主链路与删除项。删除范围在完整引用审计后确定，不为旧产品保留隐藏兼容层。
- 本次主线重构已获得开发期破坏性协议调整授权；具体 Schema / API 删除与替换仍必须在 `design.md` 中列出，并用新主链路的端到端验证覆盖。
- 保留用户当前工作区已有改动；不得用整文件还原、破坏性 Git 操作或大范围无关重构清理历史残留。
- 实施前必须形成 `design.md` 与 `implement.md`，明确文档、前台、后端兼容、数据迁移、验证和回滚边界。
- 实施只在单一职责叶子任务中进行。阶段任务仅协调依赖，不直接承接跨层大改；每个叶子任务必须有独立验收、验证命令和回滚点。
