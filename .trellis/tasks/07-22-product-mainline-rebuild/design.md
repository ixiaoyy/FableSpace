# 产品主线重构技术设计

## 1. 目标结构

FableSpace 只保留角色故事平台这一条产品主线。P0 内容与玩家身份如下：

| 故事世界 | 公开角色 | 固定 PlayerRole | 题材职责 |
|---|---|---|---|
| 1854 年宽街 | 安妮 | 乞丐 | 验证不可改写史实边界内的历史互动 |
| 雪夜封宫 | 魏观海、萧明珠 | 小太监 | 验证原创架空世界中的角色冲突与多方关系 |

青槐驿和临川大学移出 P0 公开内容，保留为待重写候选，不得继续被首页或公开发现 API 当作已完成内容。

P0 唯一玩家链路：

```text
角色发现
  -> 选择角色
  -> 查看一句故事处境与固定 PlayerRole
  -> 开场对话 / 选择
  -> 人工剧情状态机推进 + AI 角色演绎
  -> 写回关系、选择、记忆和轮次结果
  -> 回访继续
```

## 2. 核心领域合同

### StoryWorld

`StoryWorld` 是一个系统策划、审核和版本化发布的完整故事世界。它不再包含坐标、owner、密码、营业状态、创建者配置或用户发布能力。

必需概念：

- `id`
- `title`
- `summary`
- `genre`
- `publication_status`: `draft` / `published` / `archived`
- `content_version`
- 固定 `PlayerRole`
- `Character[]`
- 人工编写的章节、选择、状态转换、结局和正史边界

### Character

`Character` 必须属于一个 `StoryWorld`，并包含角色演绎需要的动机、秘密、语言、当前处境、开场对白和关系规则。`NPC` 只描述 AI 演绎方式，不作为持久化实体名。

### PlayerRole

`PlayerRole` 必须属于一个 `StoryWorld`，包含固定称谓、性别、背景、入场理由和角色可见信息。P0 每个世界只有一个，不提供全局身份目录或用户自定义 Prompt。

### PlayerStoryState

`PlayerStoryState` 按 `player_id + story_world_id` 唯一定位，保存：

- 当前 `player_role_id`
- 当前活动轮次
- 回访次数和最近回访时间
- 已完成轮次的结局摘要

未登录玩家的 `player_id` 来自服务端认可的匿名访客标识；登录后才承诺账号绑定和跨设备恢复。客户端不得自行提交或替换任意玩家 ID。

### StoryRun

每次首次进入或结局后重新开始都会创建一个 `StoryRun`。轮次包含：

- 内容版本
- 当前章节 / 节点
- 关键选择和故事标记
- 角色关系
- 私有记忆
- 可回放事件
- `active` / `completed` 状态和结局摘要

进行中的关键选择不能回退。新轮次不继承旧轮次的好感度和分支标记，只在 `PlayerStoryState` 留存旧结局摘要。

### CharacterRelationship

每个轮次按角色保存内部连续好感度、故事专属关系阶段、最近变化原因和关键关系标记。

- 自然对话只能产生有依据、受限的小幅变化。
- 人工编写的关键选择可以产生较大变化，并打开或关闭分支。
- 重复闲聊不得无成本刷出重大关系或结局。
- 前端不显示数值，只显示角色态度、关系阶段和变化原因。

## 3. 系统内容与运行时数据边界

系统故事内容应以仓库内可审查、可版本化并可校验的内容注册表维护，不再通过 owner CRUD 写入。计划中的内容层负责：

- 校验 `StoryWorld`、`Character`、`PlayerRole`、章节和选择引用完整性；
- 只向公开 API 暴露 `published` 内容；
- 保证历史固定正史与原创可变内容分层；
- 让 Git 版本和部署版本成为系统内容的发布与回滚依据。

数据库只保存玩家身份映射和运行时私有数据：`PlayerStoryState`、`StoryRun`、关系、消息、记忆和事件日志。系统内容版本号随轮次保存，避免部署后无法解释旧进度。

## 4. 故事推进与 AI 边界

运行顺序：

```text
收到预设选择或自由输入
  -> 加载 StoryWorld 内容版本与当前 StoryRun
  -> 将输入解析到允许的故事动作，或保持为普通对话
  -> 由确定性规则校验并应用章节 / 选择 / 关系变化
  -> 构建角色、PlayerRole、正史、当前状态和私有记忆上下文
  -> AI 生成角色对白
  -> 输出校验
  -> 持久化消息、事件、受限关系变化和记忆候选
```

AI 不得直接写入正史、创建新永久结局、绕过前置条件或自行确认关键状态。自由输入可以小幅影响关系；任何写回都必须经过可追踪的规则、上限和原因代码。

## 5. API 形状

新 API 只使用 `story_world_id` 和 `character_id`。计划中的公开能力：

- 获取已发布角色发现列表，响应同时携带所属故事世界摘要；
- 获取故事世界、角色和固定 PlayerRole 的入场信息；
- 开始或恢复当前故事轮次；
- 获取私有 `PlayerStoryState`；
- 向具体角色发送消息；
- 提交人工编写的选择；
- 结局后重新开始；
- 获取面向玩家的关系阶段、态度变化和回访摘要。

具体路径由后端子任务确定，但所有故事深链统一位于 `/story-worlds/...`；不得保留 `/spaces`、中文 Space 路由或 legacy alias。认证中间件负责解析登录账号或匿名访客，不接受客户端注入 owner 或任意 visitor ID。

## 6. 前端信息架构

前端只保留：

- `/`：角色发现首页；
- `/story-worlds/:storyWorldRef/characters/:characterRef`：角色入场与故事互动；
- 必要的登录 / 账号绑定表面；
- 必要的错误、加载、无内容和回访状态。

首页以安妮、魏观海、萧明珠三张真实角色卡为 P0 数据，不补虚构占位角色。故事世界作为角色卡背景呈现，不提供独立世界目录。进入角色后展示固定 PlayerRole，不提供身份选择器。

## 7. 删除边界

新主链路验收通过后，删除以下产品和代码合同：

- `Space` / `space_id` / `/spaces` / 中文 Space 路由及兼容别名；
- `lat` / `lon`、地图入口、AMap 配置、附近发现和坐标创建；
- 用户创建、店主、owner、territory、Home 和创作者管理表面；
- SillyTavern 导入、导出、角色卡解析和兼容文档；
- owner / StoryWorld 私有 LLM 配置和店主 Token 统计；
- story package、用户发布、密码访问、营业状态；
- 与角色故事 P0 无关的公开社交、public bond、rumor、relationship graph、skill pack 和旧玩法入口。

删除前必须做入口、引用、API、数据库表、配置、部署和文档审计。仅在新主链路已通过对应验证后移除旧实现，禁止通过整文件还原覆盖当前工作区已有改动。

## 8. 数据迁移与回滚

本任务不保留旧产品数据兼容。实施仍分阶段：

1. 新领域、API 和运行时表与旧实现并存开发，不让旧表结构限制新合同。
2. 两个 P0 故事与前端切换到新 API。
3. 完成编译、类型、构建和端到端验收。
4. 对旧表、配置和文件形成精确清单，并在备份与目标路径确认后执行清退。

回滚只保证代码和系统内容版本可退回，不承诺把新玩家进度反向转换为旧 Space 数据。任何实际数据库删除必须由独立迁移步骤执行并记录目标表，不得在应用启动时静默清库。

## 9. 当前工作区保护

当前工作区已有另一任务留下的首页、路由、默认故事、权威文档和规范改动。后续子任务在触碰以下文件前必须先检查完整 diff 并吸收可保留部分：

- `apps/api/src/fablespace_api/core/default_spaces.py`
- `apps/web/app/routes.ts`
- `apps/web/app/lib/web-routes.ts`
- `apps/web/app/routes/home.tsx`
- `apps/web/app/shell/product-shell.tsx`
- `AGENTS.md`
- `docs/PRODUCT_BRIEF.md`
- `docs/FABLESPACE_SPACE_PLATFORM.md`
- `docs/WHAT_NOT_TO_BUILD.md`

不得使用整文件 `git restore` / `checkout`，不得把未提交改动误判为本任务可删除的旧代码。

## 10. 子任务依赖

```text
角色故事平台合同与术语
  -> 公开产品定位文档
  -> 平台主线与不做边界
  -> Schema 与 AI 协作约束
  -> StoryWorld 后端阶段
     -> 系统故事内容模型
     -> 玩家状态持久化
     -> 匿名与账号玩家标识
     -> 系统 LLM 运行配置
     -> 故事运行时 API
  -> P0 内容阶段
     -> 安妮宽街故事
     -> 雪夜封宫故事
  -> 玩家前台阶段
     -> 角色发现与新路由
     -> 故事对话选择界面
     -> 回访关系与登录提示
  -> 旧能力清退阶段
     -> 地图与坐标
     -> 创作者与 SillyTavern
     -> Home 社交与外围能力
     -> 旧 Space 合同与路由
     -> 旧 Schema 配置与部署引用
  -> 主线集成迁移与验收
```

五个阶段任务只协调子任务状态，不直接改文件。三个合同文档任务互不混改；领域子任务中的内容模型先行；持久化、玩家标识和系统 LLM 可在合同稳定后分别实施；运行时 API 等待这些合同完成。两个故事互不混改；三个前端表面互不混改；五类旧能力分别清退。内容依赖领域合同，前端依赖稳定 API 与 P0 内容，旧能力删除必须等待对应新链路可运行，集成任务只负责最后验证。
