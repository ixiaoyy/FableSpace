# Discord in-app screenshot brainstorm for FableMap

日期：2026-04-29  
来源：用户在本轮对话提供的 Discord 客户端截图（好友/个人资料/安全与信誉/Nitro 个性化/商店/任务/发现/服务器频道/规则声明）。  
用途：作为 FableMap 当前 Trellis 任务的产品参考，不直接照搬 Discord 品牌、付费体系、社交图谱或素材。

## 与 FableMap 边界对齐

必须保留：

- FableMap 是真实地图锚定的赛博酒馆 UGC 平台。
- 店主主权优先；平台不绕过店主确认发布酒馆/NPC/故事内容。
- 不做访客好友、私信、全局在线社交、无边界社群。
- 不做平台充值、结算、抽成或 Nitro 式付费权益。
- 不做传统 RPG 战斗、等级装备、竞技排行。

因此，Discord 截图可借鉴的是信息架构、任务卡片、状态反馈、入口组织和视觉节奏，而不是社交网络、付费系统或通用社区产品方向。

## 截图可观察模式

### 1. 左侧长期导航 + 中间主任务区 + 右侧状态区

- 好友页把「添加好友」作为当前主任务，中间区域只承载一个明确动作。
- 右侧「当前活动」用于弱提示，不干扰主任务。
- 左栏保留长期入口：好友 / Nitro / 商店 / 任务 / 私信。

FableMap 转译：

- 店主/访客页面可以采用「当前主动作 + 辅助状态」结构。
- 例如 `/owner` 中间是“下一步开门/复盘”，右侧是“最近访客、通知、未处理反馈”。
- `/discover` 中间是“附近可进入酒馆”，右侧或卡片角标显示“最近传闻/回访提示”。

### 2. 个人资料与安全页：清晰的账号健康状态

- 个人资料页字段很少，编辑入口明确。
- 安全/信誉页用进度线表达账号状态，而不是堆复杂说明。

FableMap 转译：

- 可做「酒馆开门健康度」：坐标已选、门面已填、NPC 已配置、访问规则已设、LLM 可用、欢迎语/回访反馈就绪。
- 这个模式更适合 Owner Dashboard，而不是做复杂 BI。

### 3. Nitro/商店页：个性化展示与主题包陈列

- 展示风格、头像、主题、横幅、APP 图标等“个人空间装扮”。
- 商店卡片强调视觉预览、套装、价格、购买按钮。

FableMap 转译：

- 可以借鉴“酒馆门面/氛围预览”的视觉节奏，但不要做平台付费商店。
- 允许方向：店主自有素材、内置免费主题模板、酒馆门面预览、角色头像/表情资源管理。
- 不建议方向：Nitro 式订阅、付费主题包、平台商城。

### 4. 任务页：卡片式任务、明确动作、剩余时间、奖励

- 任务列表用大图卡展示：推广方、截止时间、奖励、主要动作（开始/观看/下载/接受任务）。
- 有「所有任务 / 已领取任务」Tab，右上角有推荐/筛选。
- 奖励视觉统一，任务卡有状态与 CTA。

FableMap 转译：

- 当前 `04-28-quest-system` 最值得借鉴这一组，但应改名/转译为“探索清单 / 酒馆委托 / 店主引导”，降低传统 RPG 任务感。
- 任务奖励不应是可交易货币或等级装备；更适合：纪念章、称号、解锁表情、酒馆内回访提示、店主可见完成记录。
- 任务动作应绑定 FableMap 核心链路：进入酒馆、与 NPC 对话、回访、留下 owner-visible 反馈、创建自己的真实坐标酒馆。

### 5. 发现页：APP / 服务器分流 + 类目导航

- Discord 把 APP 与服务器分成不同发现入口。
- 卡片展示图、名称、分类、简介、在线/成员数等活性信号。

FableMap 转译：

- FableMap 可以分为「发现酒馆」与「创作工具/模板」两条入口，但不做 App Store。
- 酒馆发现卡片可展示：地点类型、氛围、角色数、最近传闻、是否公开、回访状态。
- 类目不应是通用游戏/娱乐，而应是：地点类型（咖啡店/书店/便利店/学校）、氛围（安静/热闹/深夜/治愈）、城市/附近、是否适合新手。

### 6. 服务器频道与规则页：治理结构很清楚

- 新用户进入后先看到 Welcome、Rule、公告、社区链接引导、验证频道。
- 规则页将高风险行为单独列出，声明格式清晰，更新变更有要求。
- 右侧成员列表显示角色与身份层级。

FableMap 转译：

- 每个酒馆可以有「入店须知 / 店主声明」区，用于访问规则、NPC 扮演边界、AI 记忆说明。
- 对 player-as-NPC/hybrid 模式，必须明确审批、撤销、权限范围，不形成跨酒馆社交图谱。
- 可以为店主提供“规则模板”，但店主确认后才发布。

## 重点：任务系统可参考方向

### 推荐产品命名

优先考虑：

1. **探索清单**：最克制，强调引导访客理解地图和酒馆。
2. **酒馆委托**：更有赛博酒馆味道，可由店主发布轻量目标。
3. **今日引路牌**：偏发现/新手引导，避免 RPG 任务感。

暂不建议直接叫“Quest”或“任务系统”，容易滑向 RPG/等级/刷奖励。

### MVP 信息结构

可参考 Discord 任务卡，但字段转译为：

```text
卡片封面：酒馆门面 / 地点氛围 / NPC 头像组合
发起方：平台新手引导 / 某个酒馆店主 / 某 NPC
标题：例如“和深夜树洞的值班 NPC 聊一次”
目标：进入指定公开酒馆并完成 3 轮对话
截止：可选，MVP 可不做限时；若做只用于活动，不做压力运营
奖励：纪念章 / 表情 / 回访称号 / 完成记录
CTA：开始探索 / 接受委托 / 继续进度
状态：未领取 / 进行中 / 已完成 / 已放弃
```

### MVP 任务类型建议

1. **新手开门任务**（访客向）
   - 目标：进入任意公开酒馆，与任意 NPC 完成 3 轮对话。
   - 价值：验证核心闭环。

2. **街区探索任务**（访客向）
   - 目标：在同一区域拜访 2-3 家不同地点类型的酒馆。
   - 价值：强化真实地图锚定。

3. **回访记忆任务**（访客向）
   - 目标：第二天回访同一酒馆，查看 NPC 是否记得你。
   - 价值：让记忆写回变得可感知。

4. **开店准备清单**（店主向）
   - 目标：完成坐标、简介、NPC、访问规则、LLM 配置。
   - 价值：与 Owner Dashboard 形成闭环。

5. **店主可选委托**（后续，不进首版）
   - 目标：店主在单个酒馆内配置轻量目标，例如“听酒保讲一次今日传闻”。
   - 边界：只能绑定本酒馆/NPC，不能创建跨访客社交或全局排行。

### 不建议进入 MVP

- 下载/观看广告式任务。
- 货币奖励、付费任务、抽奖。
- 跨酒馆竞技排行榜。
- 访客互加好友、组队、私信协作任务。
- AI 自动生成并发布任务；若有 AI 草稿，必须店主确认。

## 推荐下一步拆法

### 方案 A（推荐）：先改 Quest PRD，不急着实现

目标：把 `04-28-quest-system/prd.md` 从传统“任务系统”重写为更符合 FableMap 的“探索清单 / 酒馆委托 MVP”。

优点：
- 风险最低，不动代码。
- 先修正方向，避免后续实现偏向传统游戏系统。
- 可以直接吸收本次截图中最强的“任务卡片”参考。

### 方案 B：在 Owner Dashboard 中先做“开门准备度”

目标：借鉴安全/信誉页的进度线，用前端聚合状态告诉店主还差什么才能开门迎客。

优点：
- 与当前 review 任务关联强。
- 不需要新增任务模型。
- Demo 观感明显提升。

### 方案 C：做发现页专题卡片

目标：借鉴 Discord 发现页的服务器卡片，把 FableMap 酒馆卡片升级为“场所入口”。

优点：
- 对访客体验直接。
- 可复用地点类型/传闻/公益酒馆素材。

风险：
- 容易滑向排行、推荐算法或传统 POI，要保持克制。

## 建议优先级

1. **P1：重写 Quest PRD 为“探索清单/酒馆委托”** —— 最贴合用户强调的“任务相关”。
2. **P1：Owner Dashboard 开门准备度** —— 可与当前 owner dashboard review 任务形成补充。
3. **P2：发现页专题卡片/筛选** —— 适合后续单独设计。
4. **P3：酒馆规则/入店须知模板** —— 等 hybrid/player-as-NPC 深化时再做。

## 可验收标准草案（Quest 方向）

- 任务/清单卡片只推动 FableMap 主链路：地图发现、进店、NPC 对话、回访、店主确认创作。
- UI 至少有“全部 / 已领取”两个视图或等价过滤。
- 每张卡有：发起方、目标、状态、CTA、奖励说明。
- 奖励不是货币、等级、装备或可交易资产。
- 不包含访客好友、私信、组队、排行榜。
- 若允许店主创建委托，委托只在单个酒馆范围内生效，且必须由店主手动发布。
