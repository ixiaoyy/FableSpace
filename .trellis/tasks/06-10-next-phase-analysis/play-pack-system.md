## 背景与问题

### 当前状态
- `PlayPackPanel` 已存在，支持玩法包注入
- `CULTIVATION_PLAY_PACK` 作为第一个玩法包
- 缺少 owner-local 玩法模板管理和导入预览

### 问题分析
1. **包数量少**: 目前只有一个玩法包
2. **管理缺失**: 无法上传、编辑、删除玩法包
3. **治理未定**: 公开市场、评分评论和安装机制需要内容审核与权限设计
4. **分享缺失**: 店主无法分享自己的玩法包

### 用户故事
```
作为 店主
我希望 能浏览项目内置或自己保存的玩法模板
以便 快速丰富我的空间内容
```

## 技术实现

### 后端 API

第一阶段不默认新增 `/play-packs` API。优先复用或组合已有能力：

| 现有能力 | 用途 |
|----------|------|
| `/api/v1/taverns/{id}/gameplays` | 保存店主确认后的 gameplay definitions |
| `/api/v1/taverns/{id}/package` | 导出不含密钥/访客数据的空间包 |
| `/api/v1/tavern-packages/import` | 导入空间包，需店主选择坐标并确认 |
| `/api/v1/taverns/{id}/skill-packs` | 启用/关闭受边界约束的能力包 |

若后续确实要做公开玩法包市场，再单独设计 API、存储、审核、版权、可见性和滥用治理。

### 数据存储

第一阶段模板可先放在前端/项目内置配置或 owner 私有配置中，不新增 `play_packs` 表。新增持久模型前必须先回答：

- 模板是 owner 私有、项目内置，还是公开市场内容？
- 模板内容是否能随空间包导出？
- 模板是否包含 NPC、WorldInfo、GameplayDefinition，还是只包含玩法定义？
- 安装预览如何证明不会携带 API Key、访客记忆或运行时会话？

## 校准补充

### 已核对事实

- 现有后端有 `/api/v1/tavern-packages/import`、`/api/v1/taverns/{id}/package`，这是空间包导入/导出，不是玩法包市场。
- 现有 skill pack 能力已有前后端边界文档；`local-rumor` 等能力包不能自动改写正史或 NPC。

### 边界修正

- “玩法包市场、评分评论、安装量排行”会引入内容审核、版权、平台治理和社交/评分系统风险，不建议作为 P3 第一阶段。
- 玩法包安装必须是店主确认后的导入/启用，不得自动发布 NPC、WorldInfo、玩法或访问规则。
- 包内容不能携带 owner API Key、访客数据、运行时记忆或 hidden prompt。

### 建议 MVP

1. 先厘清三类概念：Tavern package（空间包）、SkillPack（能力包）、GameplayDefinition template（玩法模板）。
2. 第一版只做 owner-local 玩法模板库/导入预览，确认后写入当前 Tavern 的 gameplay definitions。
3. 市场、评分、评论、公开上传暂缓，等内容审核和治理模型明确后再讨论。

### 需要确认的问题

- “玩法包”到底要复用 Tavern package，还是只包含 gameplay definitions/WorldInfo/角色模板？
- 包由平台内置、店主私有，还是允许公开给其他店主？三者权限和审核完全不同。
- 是否需要兼容 SillyTavern preset/export，还是只服务 FableMap 内部？
