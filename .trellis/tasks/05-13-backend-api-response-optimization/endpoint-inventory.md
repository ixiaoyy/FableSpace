# Endpoint Inventory — Backend API Response Optimization

Status: researched / inventory pass completed on 2026-05-13.

## Scope and method

This inventory was generated from FastAPI route decorators in:

- `backend/src/fablemap_api/core/web/app.py` app-level `/api` fallback and frontend index route.
- `backend/src/fablemap_api/core/web/router.py` legacy/core `/api/*` compatibility surface.
- `backend/src/fablemap_api/api/v1/*.py` native `/api/v1/*` route modules included by `backend/src/fablemap_api/api/v1/router.py`.
- Global error handlers in `backend/src/fablemap_api/core/web/app.py` and `backend/src/fablemap_api/app_factory.py`.
- Frontend consumers were quick-grepped in `frontend/app/lib/taverns.ts` and `frontend/app/product/services/tavernService.js`; exact dynamic-template matching is intentionally conservative.

## Global response observations

- Both app factories normalize `HTTPException` to `{"error": exc.detail}` and unexpected exceptions to `{"error": "服务暂时不可用"}`.
- FastAPI/Pydantic request validation still uses the default `{"detail": [...]}` shape; frontend readers already accept both `error` and `detail`.
- Most v1 routes are thin and delegate to `TavernApplicationService` or focused application services, but response models are uneven: some list/status endpoints use Pydantic response models while many return raw `dict[str, Any]`.
- Legacy `/api/*` routes remain active through `create_api_router(service)` and often mirror v1 functionality; do not remove or reshape them without a compatibility task.
- Public/share/visitor responses must preserve redaction: no owner API keys, hidden prompts, passwords, private visitor state, raw provider payloads, or full private transcripts.
- Binary/static/proxy responses (`/api/taverns/{id}/tts`, `/api/tts/synthesize`, `/generated/*`, `/sillytavern/*`) must stay out of generic JSON envelope rewrites.

## Risk labels

- `P0-blocker`: changing the shape can break core tavern discovery/create/enter/chat/runtime/memory/gameplay flows.
- `P1-contract`: important endpoint family with typed frontend clients, owner/admin boundaries, or persistent schema visibility.
- `P2-local`: contained utility/status endpoint or low-fanout response.
- `P3-doc-only`: documentation-only cleanup; no current route in this inventory was downgraded to this label.

## Compatibility risks found

1. **Dual API surfaces:** many flows exist on both legacy `/api/*` and native `/api/v1/*`; optimizing one surface without parity tests can break product-parity screens or older helpers.
2. **Mixed list envelopes:** list responses use named arrays (`taverns`, `homes`, `rumors`, `territories`, etc.) plus `count` inconsistently; some include no `limit/offset`.
3. **Mixed action acknowledgements:** delete/action endpoints return `{"ok": true}`, `{"success": true}`, delegated domain payloads, nullable resources, `FileResponse`, `Response`, or proxy responses. This needs endpoint-family migration, not a global sweep.
4. **Validation envelope mismatch:** `HTTPException` is normalized to `error`, but Pydantic validation emits `detail`; changing either requires frontend reader and tests.
5. **Binary/non-JSON endpoints:** TTS, generated/static, SPA index, and SillyTavern proxy routes must stay out of JSON-envelope normalization.
6. **Preview flags already have domain-specific names:** GM layer, episode export, voice greeting, and visual souvenir use `preview_only`, `applied`, `persisted`, `audio_generated`, `image_generated`, and `requires_confirmation`; a contract should standardize future additions without renaming current keys blindly.
7. **Encoding risk:** some source files contain mojibake-looking literals in console output/AST summaries. Do not rewrite Chinese error messages as part of response optimization unless verified from source encoding and tests.
