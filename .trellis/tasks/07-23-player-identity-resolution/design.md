# 强制登录玩家身份技术设计

## 1. 目标边界

本任务把现有“整站门禁 + StoryWorld API 匿名放行”调整为清晰的公开 / 私有边界：

- 首页、角色发现、已发布 Character 公开详情和固定 PlayerRole 入场信息是公开读取能力。
- 开始、恢复、推进、读取私有状态和重新开始故事是登录后能力。
- 私有请求只接受服务端已验证会话中的账号 ID，客户端不传 `player_id`。
- 未登录或会话失效时返回 `401`，并保证应用服务和数据库写入尚未发生。

本任务不新增账号系统。它复用现有 ParallelLines 票据交换、签名 HttpOnly 会话 Cookie、权限 introspection 和 `fablespace.access` 能力校验；独立部署模式仍须通过同一服务端身份解析合同提供可信账号身份，不得回退到匿名 Cookie。

## 2. 路由与门禁划分

公开范围必须使用精确路由或显式路由元数据，不得继续豁免整个 `/api/v1/story-worlds/` 前缀：

- 公开：认证状态、登录回调、首页所需角色发现、已发布角色详情。
- 私有：`runs/current`、创建轮次、消息、选择、重新开始，以及后续 PlayerStoryState、关系、记忆和结局摘要接口。

私有请求先完成会话验证和访问资格检查，再解析 `player_id` 并调用 StoryWorld 应用服务。错误边界如下：

- 无会话、无访问资格或会话过期：`401`，`Cache-Control: no-store`。
- 已登录但无权访问具体资源：沿用不泄露记录存在性的 `404` / 受控领域错误。
- 请求体中的 `player_id`：不进入协议；若后续接口意外收到同名字段，拒绝或忽略并以服务端身份为准。

## 3. 玩家身份解析

删除 StoryWorld 运行时的 `fablespace_player_id` 匿名 Cookie 生成与解析。私有路由从认证中间件已经验证并挂载的会话身份读取账号 ID，传给应用服务作为 `player_id`。

身份解析必须满足：

- 同一账号在不同设备上解析为同一 `player_id`。
- 不同账号始终解析为不同 `player_id`。
- 只有完成实时或缓存期内权限校验的会话可进入私有路由。
- 会话失效后不使用请求头、查询参数、本地存储或旧匿名 Cookie 兜底。

现有 `resolve_request_user_id` 的 standalone 兼容分支不能成为新 StoryWorld 合同；实现时应为 StoryWorld 私有路由提供“必须有可信账号”的解析函数或依赖，缺失身份即 `401`。

## 4. 前端登录门禁与安全回跳

根布局不再把整个应用树视为私密空间。公开页面先渲染；只有“进入故事”、恢复轮次和其他私有动作查询认证状态。

未登录点击进入故事：

1. 记录规范化的本站角色深链作为 `return_to`。
2. 跳转现有 ParallelLines 登录 / FableSpace 票据入口。
3. 回调建立 HttpOnly 会话后，以 `303` 返回经过校验的 `return_to`。
4. 页面重新请求服务端状态，恢复活动轮次或进入明确的新故事开始状态。

`return_to` 只接受允许的相对 `/story-worlds/...` 路径。拒绝绝对 URL、`//`、反斜杠、控制字符、非允许路由和经过多次解码后改变语义的值；无效或缺失时回到 `/`。回跳参数如需跨站票据传递，必须由现有受信任 SSO 流程签名或由服务端绑定到一次性票据，不能单独信任浏览器提交值。

## 5. 会话失效

私有 API 返回 `401` 时，前端清除缓存的访问状态并显示重新登录动作。对消息、选择、开始和重新开始等写请求：

- 不自动重试或自动重放。
- 不显示“已保存”或乐观提交成功。
- 重新登录后重新读取服务端当前轮次，再由玩家决定是否再次执行动作。

这样避免会话在服务端已提交与客户端收到响应之间产生不确定时，自动重放造成重复选择或重复轮次。

## 6. 数据与迁移

本任务不新增匿名到账号的迁移，也不读取、认领或合并旧 `fablespace_player_id` 数据。若环境已有匿名测试数据，保留原记录供后续显式清退任务审计，但新代码不得继续创建或访问这些记录。

玩家状态表结构仍使用 `player_id + story_world_id`；状态持久化任务只需要接收可信 `player_id`，不因本决策改动其领域结构。

## 7. 验证

- 未登录访问首页和公开角色详情成功。
- 未登录访问每个私有运行时路由均得到 `401`，数据库前后无新增或修改。
- 同一账号跨两个浏览器会话恢复同一状态；两个账号严格隔离。
- 伪造请求头、查询参数、请求体 `player_id` 和旧匿名 Cookie 均不能改变身份。
- 外部、协议相对、反斜杠、双重编码和非允许 `return_to` 被拒绝或回退 `/`。
- 会话过期的写请求不落库、不自动重放，重新登录后以服务端状态为准。

最小验证包括 Python 编译、定向认证 / API 真实请求、前端 typecheck 与 build；涉及页面门禁时补桌面和 360px 浏览器验收。

## 8. 回滚

回滚只恢复代码版本与既有会话门禁实现，不通过重新启用匿名 Cookie 掩盖失败。若必须重新开放匿名游玩，需要新的产品决策、独立数据合同和迁移方案。
