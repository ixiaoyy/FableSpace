# NPC 驱逐访客机制

## Goal

让 NPC 在符合自身人设、所属 Space 正史与当前关系后果时，能够主动终止一次不愉快的互动；系统展示 NPC 的最后回应后，将该访客退出当前角色所属 Space 并返回首页，使“拒绝”成为真实、可记住、可回访的角色行为，而不只是无后果台词。

## Background and Confirmed Facts

- 产品文档已把 NPC 的“拒绝”和“承担后果”列为核心动作，并要求角色依据自身欲望、秘密、语言和代价形成差异化反应。
- 当前单聊返回普通 `response`、Space 级 `visitor_state`、`affinity`、记忆和连续性候选，但没有结构化的“终止互动 / 驱逐”动作。
- 当前前端聊天工作台收到 NPC 文本后仍保持输入区可用，不会因 NPC 的拒绝台词自动退出；首页路由为 `/`。
- 当前 `VisitorState` 按 `space_id + visitor_id` 隔离，关系强度范围为 `0.0-1.0`，阶段只有 `stranger` 至 `best_friend`；关系可下降到零，但没有“不喜欢 / 敌对 / 被驱逐”状态。
- 仓库已有私有 `VisitorRelationshipProjection`，可按 Space 或 Character 节点记录单个访客的 `affinity` 与 `hostility`；既有负向规则是先把 `affinity` 降到 `0`，再把剩余负向量计入 `hostility`，可用于承载“仅驱逐者关系降级”。
- 项目禁止把自由文本 LLM 输出直接当成最终 UI、Schema 或正史；驱逐必须经过结构化、可落库、可回放、可验证的中间层。
- 游玩身份不能绕过或改写 Space 的公开 / 密码 / 私密访问控制；驱逐也不应暗中改写 Space 全局访问规则。

## Requirements

- R1. Space 内所有 NPC 都拥有把当前访客赶出整个 Space 的同等资格，不因主人、守门人、管理者或普通角色身份而分层；实际触发仍必须基于本轮互动、既有私有关系与角色设定，并形成经系统批准的结构化驱逐动作。
- R2. 驱逐发生时，客户端立即锁定当前聊天输入并离开当前 Space、返回首页；首页显示可手动关闭的持久驱逐结果卡，确保用户能够完整阅读 NPC 的最后回应。
- R3. 客户端不得通过匹配“滚 / 走 / 出去”等自然语言猜测驱逐；服务端必须返回明确的结构化结果。
- R4. 驱逐结果必须按当前访客边界持久化为 Space 级事件，并能在聊天记录、私有记忆或后续回访状态中复核；由执行驱逐的 NPC 单独承担关系后果。
- R5. 驱逐不得公开暴露访客私密状态，不得形成公开黑名单、社交图谱或平台级惩罚。
- R6. 驱逐不得改写 Space 正史、角色卡、坐标、全局营业状态或既有访问控制。
- R7. 不同 NPC 应能依据各自人设采用不同的警告、容忍度、驱逐台词和回访态度，不能套用统一辱骂模板。
- R8. 模型失败、降级回复或结构化结果不完整时不得误触发驱逐。
- R9. 一般反感或互动不愉快时，NPC 必须先给出一次明确、角色内的警告；只有访客在警告后继续越界，才允许驱逐。
- R10. 严重越界行为允许 NPC 不经警告立即驱逐；一般越界与严重越界的判定必须结构化并可复核，不能只依赖模型自由文本。
- R11. 警告必须形成当前访客私有、可持久化的状态或事件，后续触发驱逐时能够证明警告已经发生；不能依赖单次 Prompt 临时记忆。
- R12. 第一版不设置驱逐后的进入冷却；访客返回首页后可以立即重新进入同一个 Space，服务端不得因刚发生驱逐而拒绝入场。
- R13. 驱逐只更新执行 NPC 对该访客的 Character 级私有关系投影：负向效果先降低 `affinity`，降至 `0` 后再增加 `hostility`；访客立即重新进入不会自动减轻或清除该后果，执行者不能把访客当作从未发生过冲突。
- R14. 驱逐后的 `hostility` 只能在执行 NPC 明确接受访客道歉，或确认访客完成符合角色逻辑的补救行为时降低；每次被接受的修复降低一档，重新进入、时间经过、普通寒暄或刷礼貌消息均不自动恢复。
- R15. Space 内其他 NPC 不因驱逐事件自动降低关系；他们可以得知该事件，但必须依据各自人设独立选择赞同、同情、质疑或无视。
- R16. 一次驱逐会消费并清除促成该次驱逐的有效警告；访客重新进入时不携带“下一次普通越界可直接驱逐”的警告状态，但执行 NPC 的关系降级与 `hostility` 完整保留。
- R17. 重新进入后的新一般越界必须重新产生一次明确警告，之后继续越界才可再次驱逐；新的严重越界仍可立即驱逐。
- R18. 警告后的普通驱逐使执行 NPC 与访客的关系降低一个阶段；严重越界导致的立即驱逐降低两个阶段。
- R19. 当执行 NPC 的 `affinity` 已不足以承受阶段降级时，剩余负向效果转为该 Character 级投影的 `hostility`；重复驱逐继续累积 `hostility`。
- R20. 驱逐的关系惩罚由系统依据驱逐类型确定，LLM 只能提出带原因类别的警告或驱逐动作，不能自由填写关系扣减数值。
- R21. 道歉或补救是否被 NPC 接受必须形成结构化、可复核的修复结果；LLM 不能通过普通友好回复暗中清除 `hostility`。
- R22. `hostility` 清零前，普通正向互动不得继续累积 `affinity`；执行 NPC 在访客重新进入后应按自身人设提供至少一种可能的道歉或补救方向，但不保证立即接受。
- R23. 首页驱逐结果卡必须包含执行 NPC、所属 Space、完整角色化最后回应和关系已变化的用户可理解提示，并保持到用户主动关闭；不得只使用自动消失的 Toast。
- R24. 首页驱逐结果卡不得暴露内部原因代码、Prompt、模型判定过程或其他访客私密数据。
- R25. 第一版只改造单 NPC 对话链路，包括私聊，以及公共频道中明确 `@` 某个 NPC、实际调用单聊接口的定向对话。
- R26. 第一版不改造多 NPC 自动接话的群聊链路；`/group-chat` 不返回结构化警告、驱逐或修复结果，群聊自然语言中即使出现“离开”等文字也不得触发客户端退出。

## Acceptance Criteria

- [ ] AC1. 满足最终驱逐条件时，聊天输入立即锁定且路由直接返回首页；用户能在首页持久结果卡中完整看到 NPC 的角色内最后回应。
- [ ] AC2. 普通拒绝、提到“离开”的叙事文本、模型降级和接口错误不会触发页面退出。
- [ ] AC3. 驱逐事件具有明确的 Space、访客、执行 NPC、原因类别和时间，可持久化并回放；最终字段需在设计阶段对照 Schema 确认。
- [ ] AC4. 一个访客的驱逐状态不会泄露或错误作用到其他访客，也不会改变 Space 的全局访问配置。
- [ ] AC5. 第一版中，被驱逐访客返回首页后可以立即重新进入同一个 Space，不出现倒计时、临时封禁或冷却拦截。
- [ ] AC6. 单 NPC 私聊和公共频道中的定向 `@NPC` 对话支持相同的驱逐规则，且任一 Space NPC 都具有同等的整 Space 驱逐资格。
- [ ] AC7. 手机、窄屏、键盘操作和 reduced-motion 场景仍能理解最后回应与退出结果；结果卡可聚焦、可手动关闭且不会自动消失。
- [ ] AC8. 一般反感的首次警告不会退出 Space；同一有效警告后的继续越界可以触发驱逐。
- [ ] AC9. 被判定为严重越界时可以直接驱逐，并记录跳过警告的原因类别。
- [ ] AC10. 刷新页面、切换 NPC 或切换聊天频道不会意外清除仍然有效的警告状态。
- [ ] AC11. 访客重新进入后，NPC 的开场态度与 Prompt 上下文能够反映上次驱逐、关系降级和持续反感；单纯重新进入不会恢复关系。
- [ ] AC12. 重复退出和重新进入不能重置或减轻驱逐造成的关系后果。
- [ ] AC13. 驱逐后只有执行 NPC 的 Character 级 `VisitorRelationshipProjection` 发生负向变化；其他 NPC 的 `affinity` / `hostility` 保持不变，但其 Prompt 可以获得该 Space 级事件并产生角色化反应。
- [ ] AC14. 驱逐后重新进入时，旧警告已清除但执行 NPC 的负向关系仍在；新的普通越界不会跳过警告直接驱逐。
- [ ] AC15. 普通驱逐确定性降低一个关系阶段，严重驱逐确定性降低两个关系阶段；相同前置状态与驱逐类型得到相同结果。
- [ ] AC16. 关系降至最低后继续发生的负向量写入执行 NPC 的 `hostility`，不会错误写入其他 NPC 或 Space 的共享好感度。
- [ ] AC17. 重新进入、等待或普通礼貌聊天不会降低 `hostility`；只有执行 NPC 接受的道歉或补救结果会确定性降低一档。
- [ ] AC18. `hostility` 大于零时普通正向互动不会提升 `affinity`；清零后才恢复现有正向关系增长逻辑。
- [ ] AC19. 访客重新进入后能从执行 NPC 的角色化回应中获知至少一种可能的修复方向，同时系统不承诺该 NPC 必须接受第一次尝试。

## Out of Scope

- 平台级账号封禁、公开黑名单或跨 Space 惩罚。
- 让运行时 AI 修改 Space 的公开 / 密码 / 私密访问规则。
- 把驱逐扩展成战斗、生命值、阵营声望或等级系统。
- 用固定敏感词表直接决定角色是否驱逐访客。
- 第一版的驱逐冷却、首页倒计时或临时禁止进入；可作为未来增强重新规划。
- 多 NPC 自动接话的群聊驱逐、群体反应、驱逐仲裁或撤销；可在单聊机制稳定后单独规划。
