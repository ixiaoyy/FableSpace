# FableMap AI-Native 世界编排架构草案 v0.1

## 文档目的

这份文档用于把近期关于 AI 技术趋势、世界模型、生成式交互体验与 FableMap 当前路线的讨论，收束为一份可被共享协作、任务拆解与后续实现直接引用的架构草案。

它不试图推翻当前 FableMap 的 2D 地图、结构化 `world`、写回协议与时间褶皱方向，而是回答一个更关键的问题：

> 在保持“永不坍缩的数字神话底座”前提下，FableMap 如何升级成一个更符合 AI 时代的世界系统？

---

## 一句话结论

> **FableMap 不应把 AI 用来替代世界结构，而应把 AI 提升为世界编排、镜头切换、长期记忆与城市人格的核心引擎。**

换句话说：

- 空间层继续保持确定性
- 世界状态继续保持结构化
- 视觉主舞台继续保持可复现渲染
- AI 主要负责解释、编排、演化与回应

这比“全 AI 视频流主世界”更慢炫，但更适合形成长期产品护城河。

---

## 背景判断

近期以世界模型、实时视频生成与互动生成世界为代表的技术趋势，证明了 AI 在“短时沉浸感”上正在快速进步。

但这些方向也暴露出对 FableMap 极其关键的结构性短板：

1. 长时一致性不足
2. 世界状态不可稳定持久化
3. 玩家痕迹难以可靠附着
4. 跨时间、跨玩家、跨会话的一致世界难以成立

而 FableMap 当前仓库已经明确押注的能力，恰好是这些系统最不稳定的部分：

- 真实地图骨架
- 结构化 `world` 数据
- 玩家写回协议
- 时间深度与历史层
- 浏览器 2D 主舞台
- 可持续演化的世界关系

因此，项目不应盲目追逐“端到端视频式世界生成”，而应在现有底座上升级成 **AI-native world orchestration** 路线。

---

## 核心判断：什么该稳，什么该交给 AI

### 必须保持稳定的层

以下层是 FableMap 的不可坍缩底座：

1. **Reality Kernel**
   - OSM / Overpass 真值
   - 坐标、道路、POI、zone、route 稳定锚点
   - 切片边界
   - 世界对象 ID

2. **Structured World State**
   - `world.json` 及其可演化结构
   - 写回事件
   - 地点状态
   - 玩家状态
   - 最近回声与归档历史

3. **Deterministic Surface Mapping**
   - `fantasy_type -> asset key`
   - `vibe_profile -> lens / pack`
   - tile / icon / layer 的可寻址映射
   - 前端主舞台的稳定渲染结果

### 应交给 AI 增强的层

以下层最适合成为 AI 的主战场：

1. **语义解释**
   - 现实地点如何被翻译成异世界语义
   - 一个地点在不同镜头下的意义差异

2. **世界编排**
   - 此时此地应触发何种反馈、事件、广播与暗示
   - 哪些内容该被强调、延迟、隐藏或唤醒

3. **长期记忆生成与检索**
   - 玩家与地点之间形成了什么关系
   - 哪些历史片段应当再次浮现
   - 城市如何“记住”某个玩家

4. **局部生成式表现**
   - scene capsule
   - 分享卡片
   - 地点插图
   - 任务微演出
   - 音景与短时氛围层

## 总体架构：四层模型

### 第一层：Reality Kernel（现实内核）

这是项目最传统、也最不该被替换的一层。

职责：

- 使用真实地理数据定义空间骨架
- 为所有地点、区域、路线建立稳定锚点
- 为写回、时间褶皱、资源映射提供对象基座
- 为多镜头解释提供同一份现实底图

设计原则：

- 这层优先追求可验证、可缓存、可重建
- 不在这层引入重生成幻觉
- 同一 slice 的核心结构必须可复现

### 第二层：Semantic Runtime（语义运行时）

这层负责把“现实对象”翻译成“世界意义”。

输入：

- OSM / Overpass 数据
- 区域上下文
- 历史片段
- 玩家状态
- 当前世界镜头

输出建议：

- `fantasy_type`
- `mood_signature`
- `faction_pressure`
- `memory_hooks`
- `possible_events`
- `lens_affinity`
- `time_fold_candidates`

设计原则：

- AI 输出优先结构化字段，而不是只生成散文
- 文本描述属于结果之一，但不是唯一结果
- 同一地点可拥有多个可计算解释，而非一个固定标签

### 第三层：World Orchestrator（世界编排器）

这是未来最值得投入的新核心层。

它不是地图生成器，而是一个持续读取世界状态并生成“下一步世界反应”的编排系统。

主要职责：

- 根据玩家行为重排地点优先级
- 触发广播、异常、任务苗头与回声显影
- 结合动态信号调整世界气氛与事件概率
- 决定默认显示哪个世界镜头
- 判断哪些变化沉淀为私人记忆，哪些进入公共层

输入：

- 当前切片世界状态
- 玩家状态与最近行动
- 写回事件流
- 时间、天气、拥堵、人流等现实代理信号
- 治理规则与可见性边界

输出：

- 结构化事件建议
- 地点排序与高亮建议
- 电台 / 播报 / 回声建议
- 镜头切换建议
- 精灵 / collectable / anomaly 刷新建议
- 后续伏笔与回访钩子

### 第四层：Generative Surface（生成式表现层）

这是最适合吸收前沿 AI 能力的一层，但它必须从属于前三层。

适合承载：

- 地点故事卡
- scene capsule
- 动态音景
- 历史褶皱切入时的短时演出
- NPC 头像与局部插画
- 可分享的视觉摘要

设计原则：

- 这层不承载世界真相
- 允许局部生成，避免全局状态依赖于生成结果
- 即使生成结果失败，也不能破坏主世界一致性

## 未来最值得押注的五个 AI 方向

## 1. World Memory Graph

目标：把玩家、地点、区域、回声、历史片段、阵营关系组织为可检索、可总结、可触发的长期记忆图谱。

价值：

- 强化长期连续性
- 支持个性化叙事回访
- 为写回、历史层、共创层提供统一记忆索引

适合作为后续 `P5 / P4 / D3` 之间的桥梁。

## 2. Action-to-Meaning Compiler

目标：不只记录玩家做了什么，还把行为模式编译成高阶世界含义。

例子：

- 连续深夜经过同一路线 -> 夜行者倾向
- 长期在治愈类地点停留 -> 修复型人格痕迹
- 高频观察边缘区域 -> 城市边界感知者

价值：

- 让世界真正“理解玩家”
- 为个性化镜头、任务、回声提供依据
- 为城市人格代理提供长期认知输入

## 3. Lens Engine

目标：把当前模式切换升级成统一的世界镜头系统。

每个 lens 应至少定义：

- 解释规则
- 资源包偏好
- 文案口吻
- 可见性规则
- 事件放大 / 抑制规则
- collectable / anomaly / spirit 分布偏好

价值：

- 让同一地点拥有多重可计算解释权
- 让视觉、叙事、交互、状态演化同步切换
- 为长期双主线体验提供统一机制

## 4. Personal Myth Layer

目标：在公共世界之上叠加私人神话层。

内容包括：

- 私人回声
- 私人任务
- 私人地点误读
- 私人精灵与收藏物
- 私人城市情绪路径

价值：

- 增强留存
- 增强用户与城市的关系密度
- 避免所有世界意义都被公共层稀释

## 5. Generative Scene Capsules

目标：不生成整个世界，只生成关键时刻的小型体验胶囊。

例如：

- 接近某地点时触发的 6~12 秒短时演出
- 切入历史层时生成的一张时间裂隙画面
- 完成写回后生成的地点传说卡
- 城市给玩家的梦境式回信

价值：

- 成本更可控
- 稳定性更高
- 更适合与确定性地图主舞台共存

## 产品方向升级：从地图产品到世界操作系统

FableMap 不应停留在“把地图转成奇幻说明”的工具型定位。

更有远见的产品形态应是：

> **现实城市的神话操作系统**

它至少应具备四种能力：

1. **解释城市**：把现实结构翻译成第二世界语法
2. **记住玩家**：保存人与地点的长期关系
3. **编排遭遇**：在合适时机让世界回应玩家
4. **生成镜头**：以不同视角重新揭示同一座城市

这意味着项目的竞争力不应来自“一次生成能力”，而应来自：

- 世界记忆厚度
- 玩家痕迹积累
- 可计算镜头系统
- 可持续演化的共创层
- 真实空间约束下的长期城市人格

---

## 对现有路线的影响

## 不应改变的部分

以下主线继续成立：

- 2D 稳定地图骨架
- 结构化 `world schema`
- 写回协议与治理边界
- 时间褶皱与历史层
- `tile / icon / pack` 资源化路径
- 浏览器 2D 主舞台

## 应追加的新主线

建议新增并逐步正式化以下方向：

1. `AIO1` · 世界编排器框架
2. `AIO2` · 镜头引擎 / lens schema
3. `AIO3` · 世界记忆图谱
4. `AIO4` · 行为到意义编译器
5. `AIO5` · 城市人格代理
6. `AIO6` · 生成式 scene capsule 规范

这些任务都不应替代现有主线，而应作为其上层升级。

## 分阶段建议

### Phase 0：保持当前主世界底座稳定

继续推进：

- 写回协议闭环
- 资源化地图接入
- 共创主线收束
- 时间褶皱协议的结构化落地准备

### Phase 1：把 AI 从“前处理生成器”升级为“中途编排器”

建议最先试点：

- lens 选择建议
- world feedback 编排
- POI 强调排序
- 事件伏笔生成
- 玩家回访钩子生成

### Phase 2：引入长期记忆系统

优先方向：

- world memory graph
- 行为模式总结
- 地点关系回访
- 私人神话层

### Phase 3：引入局部生成式表现层

在不破坏主舞台的前提下，逐步加入：

- scene capsules
- 历史切入短演出
- 地点视觉卡片
- 音景与电台增强

---

## 对共享任务拆解的指导原则

当把这条路线拆进共享任务时，应遵守以下原则：

1. **优先拆协议，再拆实现**
   - 先定义 schema、边界、输入输出
   - 再进入 API、前端、存储与渲染接入

2. **优先拆编排层，不先拆重生成层**
   - 先做 lens engine / orchestrator / memory graph
   - 不优先立项“AI 自动生成整个主世界画面”

3. **确保 AI 输出可落库、可缓存、可回放**
   - 优先结构化结果
   - 避免只能一次性消费的 prompt 幻觉

4. **所有新能力都要挂回真实空间锚点**
   - 与 `poi / zone / route / slice` 绑定
   - 不脱离现实地理骨架独立漂浮

---

## 明确不做什么

在当前阶段，本文明确不建议把以下方向作为主线：

- 全 AI 视频流主世界
- 逐帧生成地图主体
- 无锚点的自由生成空间
- 用生成画面替代结构化世界状态
- 为追求“更像 AI”而削弱确定性地图底座

---

## 结论

FableMap 未来最重要的升级，不是让 AI 直接创造整座城市，而是让 AI 学会：

- 如何解释这座城市
- 如何记住来到这里的人
- 如何在正确时机回应他们
- 如何在不破坏现实骨架的前提下，展开多重神话镜头

因此，项目的中长期方向应正式收束为：

> **确定性 2D 地理底座 + 结构化世界状态 + AI 世界编排层 + 长期世界记忆层 + 局部生成式表现层**

这条路线既承认当前 AI 技术的能力边界，也最大化利用了其在解释、规划、编排、记忆和表现增强方面的真实优势。
