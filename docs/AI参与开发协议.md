# FableMap AI 参与开发协议 v0.1

## 目标

这份协议只解决一件事：

> 允许不同开发者使用不同 AI，但所有产出必须遵守同一套项目协议、边界和验证规则。

我们不统一模型，不统一提示词，不统一工具；
我们统一的是：**输入边界、输出格式、验证流程和合并标准**。

## 项目公约

- **协议先于代码**
- **人负责，AI 辅助**
- **结果统一，工具自由**
- **局部内容不得污染全局规则**
- **所有 AI 产出默认是候选草稿，不是最终事实**

## AI 在项目中的角色

### AI 可以参与

- 需求分析
- 方案拆解
- 文档草稿
- 协议草稿
- 代码实现草稿
- 测试用例草稿
- 重构建议
- Review 辅助

### AI 不能替代

- 协议最终裁定
- 架构边界决策
- 合并审批
- 兼容性背书
- 对隐含需求的主观扩写

一句话：**AI 是协作者，不是决策者。**

## 当前协议源头

在更细分的协议文件落地前，AI 参与开发时至少必须遵守：

- `README.md`
- `docs/INDEX.md`
- `docs/PRODUCT_BRIEF.md`
- `docs/FABLEMAP_TAVERN_PLATFORM.md`
- `docs/ARCHITECTURE.md`
- `docs/WORLD_SCHEMA.md`
- `docs/WHAT_NOT_TO_BUILD.md`
- `.trellis/workflow.md`
- `.trellis/spec/`
- 当前 `.trellis/tasks/<task>/prd.md`、`task.json`、`implement.jsonl`、`check.jsonl`
- 与本次改动直接相关的 `.trellis/tasks/<task>/` 记录或规范文件

如果 AI 输出与这些文档冲突，必须以文档为准，而不是以 AI 输出为准。

## 统一开发约束

### 1. 统一命名

AI 生成的内容必须遵守项目既有命名，不允许：

- 同一概念多个名字
- 未定义的中英文混用
- AI 自创缩写
- 自发引入新的语义标签体系

### 2. 统一目录

AI 生成的产出必须进入正确目录：

- 文档 -> `docs/`
- 代码 -> 对应实现目录
- 测试 -> `tests/`
- 区域内容 -> 约定的区域内容目录

不能让 AI 凭感觉决定“放哪都行”。

### 3. 统一 Schema

AI 不允许：

- 擅自新增字段
- 擅自更改字段类型
- 擅自更改枚举语义
- 擅自更改必填/可选含义

如果确实需要改协议，必须先走协议变更流程。

## 核心工作模式：Task First

本项目不接受“公共让 AI 自由发挥”的开发方式。

正确顺序是：

1. 先从 `.trellis/tasks/` 中选任务、补任务或创建新任务
2. 使用 Trellis `start` / `before-dev` / `check` / `finish-work` 等技能或等价脚本建立上下文
3. 对中高风险任务先在任务目录留下 PRD、认领说明、研究记录或 implementation plan
4. 再给 AI 明确边界
5. 再让 AI 辅助产出
6. 最后人工验证与提交

当前仓库内统一任务入口为：

- `.trellis/tasks/`
- `.trellis/workspace/<developer>/`
- `.trellis/spec/`

旧共享任务清单、`docs/claims/` 与 `docs/changes/` 已于 2026-05-12 清理出主线仓库入口。新的任务领取、状态流转、验收记录必须写入 Trellis；不要再新增旧式 claim/change 文档。

### Trellis 技能使用要求

- 开始新会话或切换任务时，先使用 Trellis `start` skill 或等价脚本读取当前任务、开发者身份、git 状态和 active tasks。
- 开始写代码前，必须使用 `before-dev` skill 或等价流程读取 `.trellis/spec/` 中与本次改动相关的 backend / frontend / guides 规范。
- 新功能、bug 修复、重构级改动优先在 `.trellis/tasks/<task>/` 下维护 `prd.md`、`implementation-plan.md`、`implement.jsonl`、`check.jsonl` 或验收记录。
- 跨层 API / Schema / 数据流变更必须把对应规范沉淀回 `.trellis/spec/` 或权威 `docs/`，不能只停留在聊天记录。
- 完成前必须按 Trellis `check` / `finish-work` 思路运行真实验证，并在汇报中列出命令与结果。

### 每个任务卡至少包含

- 目标
- 范围
- 输入协议
- 输出物
- 不可触碰边界
- 验收标准
- 验证方式

### 敏捷切片补充原则

在遵守协议与认领规则的前提下，默认采用**小步、单切片、轻前置**的执行方式：

- 一次只推进一个可见切片：同一轮尽量只做一个用户能直接感知的变化
- 优先最小闭环：先做可展示、可测试、可收口的 MVP，再决定是否扩展
- 认领说明写到“刚好够用”：说明当前切片目标、边界、验证方式即可，不要求把长期机制一次写满
- 不鼓励同一轮并行铺开多个未来层：如果任务过大，应先拆成更小子切片再进入实现
- 文档以真实进度为准：先完成最小实现与验证，再补 change 文档同步真实结果

## 开发前认领规则

为了避免重复开发、边界冲突和 AI 自由扩写，中高风险任务在进入实现前，必须先在 Trellis 中完成认领。

认领位置只接受：

- `.trellis/tasks/<task>/task.json`：`assignee`、`status`、`current_phase`、`scope`、`notes`
- `.trellis/tasks/<task>/prd.md`：目标、范围、验收标准、不可改边界
- `.trellis/tasks/<task>/implementation-plan.md` 或同目录实现记录：执行计划、验证记录
- `.trellis/tasks/<task>/implement.jsonl` / `check.jsonl`：需要注入给实现和检查阶段的上下文

不要再使用旧式 `docs/claims/*.md` 作为认领说明；如需历史依据，写入当前 Trellis 任务的 `prd.md`、`implementation-plan.md` 或 `task.json.notes`。

### 强制适用场景

- 协议变更
- 新模块开发
- 跨文件功能开发
- 区域内容认领
- 可能与其他开发者冲突的并行改动

### 可豁免场景

- typo 修复
- 纯格式调整
- 小范围文档润色
- 不改变边界的轻量测试修正

### Trellis 认领说明至少应声明

- 模块名 / 区域名
- 负责人
- 改动类型
- 目标
- 可修改范围
- 明确不改范围
- 依据的协议文档
- 预期产出
- 验证方式
- 当前状态

认领说明的作用是防止重复开发和边界失控；
它不能替代每次 commit 对实际改动的文档说明。

### 旧共享任务迁移规则

- 旧共享任务清单不再保留在 `docs/` 中；任务权威状态以 `.trellis/tasks/` 为准。
- 如果从旧聊天、旧提交或外部记录发现未闭环事项，必须先创建/更新 Trellis 任务并补 `prd.md`、`task.json` 状态。
- 不要为了恢复旧索引而重新创建 `docs/AI_SHARED_TASKLIST.md`、`docs/claims/` 或 `docs/changes/`。

## 每次 AI 参与都必须带的输入声明

开发者在调用 AI 前，至少要向 AI 明确这些信息：

- 当前任务目标是什么
- 允许修改哪些文件
- 不允许修改哪些文件
- 依据哪些协议文档或版本
- 哪些内容不能假设
- 必须完成哪些验证

没有边界声明的 AI 参与，视为不合规协作方式。

## 变更类型必须分离

### 协议变更

包括：

- Schema 变更
- 映射标准变更
- 命名规范变更
- H3 组织方式变更

### 功能变更

包括：

- 引擎实现
- 工具实现
- 客户端功能
- 测试与校验逻辑

### 区域内容变更

包括：

- 局部区域设定
- H3 网格内容补充
- 示例世界数据更新

规则：**一个 PR 尽量只做一类变更。**

## AI 生成产出必须满足的 6 条硬规则

### 规则 1：不可修改无关文件

只能改任务允许范围内的文件。

### 规则 2：不可虚构依赖

不允许引用不存在的模块、包、接口或数据。

### 规则 3：不可绕过协议

协议未定义的内容，AI 不能自行拍板后直接落地。

### 规则 4：不可跳过验证

AI 生成后必须做最小验证，按改动类型选择：

- lint
- test
- schema validate
- CLI smoke test
- preview load check

### 规则 5：不可省略变更说明

每次提交必须说明，而且**每次 commit 都必须附带文档说明**。

仅有 commit message 不足以满足本协议；
必须在仓库中留下可追踪的文档信息。

可接受形式包括：

- 更新相关设计文档
- 更新模块说明文档
- 新增一份本次改动说明文档

每次 commit 对应的文档说明至少应包含：

- 为什么改
- 改了什么
- 影响了哪些文件或模块
- 没改什么
- 是否涉及协议 / Schema / 命名变更
- 做了哪些验证
- 风险点是什么

### 规则 6：不可把 AI 输出视为真相

AI 输出默认只是候选实现，必须经人工确认。

## 风险等级

### A 类：低风险

适合 AI 高参与：

- 文档整理
- 测试补全
- 小工具脚本
- 纯配置补录

### B 类：中风险

适合 AI 辅助、人主导：

- 模块内逻辑新增
- 转义规则实现
- 数据转换流程
- 客户端局部功能

### C 类：高风险

AI 只能辅助，不可直接主导：

- 协议变更
- 核心架构变更
- 多模块联动重构
- 全局命名调整

## PR 要求

如果本次工作有 AI 参与，PR 必须说明：

- AI 是否参与
- AI 参与了哪一段：分析 / 草稿 / 实现 / 测试 / 文档
- 依据了哪些协议文档
- 对应的模块认领说明在哪里（如适用）
- 对应的 commit 文档说明在哪里
- 自己做了哪些人工确认
- 跑了哪些验证

## Reviewer 检查重点

Reviewer 不判断“像不像 AI 写的”，只判断是否符合协议：

- 是否越界修改
- 是否偷偷改 Schema
- 是否新增未定义语义
- 是否伪造依赖或接口
- 是否完成验证
- 是否把局部规则污染成全局规则
- 是否遵守模块认领说明

## 区域内容的特别规则

区域贡献应逐步统一到 H3 网格。

每个区域内容提交至少应声明：

- `h3_index`
- `region_name`
- `protocol_version`
- `owner`
- `status`

区域贡献者不能：

- 修改全局协议
- 修改全局映射标准
- 为本地区域定义专属字段结构
- 让局部命名覆盖全局术语

区域内容只能是：

- 对全局规则的补充
- 对本区域的细化
- 对特殊场景的受控覆盖

## 明确禁止的行为

- 直接让 AI 自由生成后原样提交
- 未定义协议就先写实现
- 在功能 PR 中夹带协议修改
- AI 擅自新增字段、枚举、目录
- 提交未经验证的 AI 代码
- 未认领就直接开展中高风险模块开发
- 提交没有附带文档说明的 commit
- 提交“我没看懂但 AI 说可以”的内容
- 在未声明边界时让 AI 修改整个仓库

## 最终原则

本项目允许任何 AI 参与开发，但不允许任何 AI 绕过协议。

开发者可以自由选择 AI 工具；
但提交结果必须遵守统一的世界协议、命名规范、目录规范和验证流程。

协议变更优先于实现变更，局部内容不得污染全局规则。
