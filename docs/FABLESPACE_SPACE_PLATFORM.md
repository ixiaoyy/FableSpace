# FableSpace 空间平台主线

本文档定义 FableSpace 当前 P0 主线。产品简报见 [PRODUCT_BRIEF.md](PRODUCT_BRIEF.md)，数据结构见 [WORLD_SCHEMA.md](WORLD_SCHEMA.md)，生产运行与权限联动见 [DEPLOYMENT.md](DEPLOYMENT.md)。

## 主线定位

FableSpace 不是传统地图 App，也不是 AI 聊天室列表，而是**真实坐标锚定的角色故事平台**。

每个 Space 都绑定真实地理位置，并拥有稳定的主题、正史、角色、当前事件、访问边界与连续性状态。玩家从想见的角色进入完整世界，由 NPC 以自身立场推动选择、线索、关系和后果，并在回访中延续记忆。

前台主入口采用“先选择游玩身份，再选择想见的角色”的顺序。角色优先不是展示层调整，而是当前产品主线；NPC 仍必须属于一个完整 Space，真实坐标、世界正史、访问控制、场景、后果和记忆边界都不能降级。

## P0 闭环

```text
选择游玩身份与自声明性别
  -> 角色优先发现，选择想见的 NPC
  -> 查看角色所属空间、真实位置语境与访问状态
  -> 进入角色所属的完整空间
  -> NPC 结合身份、角色设定和空间场景主动引导
  -> 对话 / 选择 / 任务 / 创作
  -> 写回访客状态、记忆或状态卡
  -> 回访时延续关系
```

## 核心角色

| 角色 | 产品职责 |
|------|----------|
| 玩家 / 探索者 | 选择私有身份，发现角色，进入完整世界，作出选择并回访。 |
| NPC | 以独立欲望、秘密、语言和代价推动所属世界的当前事件。 |
| 系统内容 | 提供经过评审、可版本化、可迁移的故事世界与规则回退。 |

店主 / 创作者供给侧是既有兼容面，不是当前 P0 产品角色。`beggar`（乞丐）是探索者选择的访客私有**游玩身份**，不授予任何权限，也不替代 Explorer 的平台身份。

## 游玩身份

游玩身份为 NPC 提供稳定、可验证的访客扮演上下文。首版规则：

- 唯一可选身份为 `play_identity_id=beggar`，前台统一展示为“乞丐”，不单独展示“古代”；“古代”是服务端版本化身份定义中的固有属性，不新增客户端可编辑 Prompt。
- 身份选择前台采用“身份目录 -> 详情与附属设置确认”的稳定两步结构；增加身份只扩展或滚动目录，不得把性别设置和确认操作继续向页面下方推移。仅在目录实际包含多个可浏览预览项时展示搜索或筛选。
- 前台可以预览尚未开放的候选身份，但必须以锁定态明确标记为不可选择；当前只有“乞丐”可进入确认步骤，其他预览项不得生成 `play_identity_id`、不得进入请求，也不得暗示已经具备对应玩法能力。
- 玩家同时自声明性别；当前前台只开放 `male` / `female`，平台不得推断。自定义称谓可以作为锁定预览出现，但在后端协议和持久化支持前不得提交或假装生效。
- 身份和性别只进入当前访客的运行时 Prompt、私有状态、记忆与回访链路，不进入公开发现、Space 分享 payload 或其他访客资料。
- NPC 必须从已发布的人设、所属空间正史和当前事件出发形成自己的反应；古代、现代等认知差异直接复用既有 `system_prompt`、`personality`、`scenario`、`description` 与 Space 场景判断，不新增平行的角色时代字段，不批量改写角色卡。
- 游玩身份不能绕过 `public` / `password` / `private`、营业状态或 owner 权限边界。

## 空间是什么

空间是挂在真实坐标上的可进入体验，也是一座拥有独立正史、规则、居民和当前事件的故事世界。真实坐标是这座世界在现实地图上的锚点和入口，不限制内部采用古代、现代、志怪或其他经过评审的叙事设定。它至少包含：

- `lat/lon` 真实坐标。
- 稳定的名称、描述、主题或氛围。
- 访问规则：公开、密码或私密。
- 一个或多个 NPC，格式优先兼容 SillyTavern。
- 可选世界知识、玩法定义、技能包和记忆机制。
- 明确的 NPC 运行后端与连续性策略。

不同 Space 之间默认不共享权威、物品、能力、常识或世界规则；跨世界互动只通过访客身份、对话与私有记忆建立，不自动改写任一世界的正史。空间可以承载陪伴、线索调查、选择与小委托，但不应变成传统战斗 RPG、通用社交群或导航 POI。

### 系统默认内容策略

系统默认 Space 的职责是直接证明“角色属于完整世界、对话能够推进故事”，而不是承担功能教程。首批仅公开三座平台原创世界：`story_palace_snow_edict`、`story_ghost_foxfire_debt`、`story_campus_last_class`，分别覆盖皇宫权谋、古代鬼怪和现代校园；每座先放 2 个关系鲜明的角色，验证后再扩展。

首页另设一个经过评审的单角色历史试验：`history_broad_street_water_1854`。玩家完成“乞丐”身份选择后，首页只展示原创小女孩安妮，并用带 `character_ref` 的 Space 入口直接进入她的私聊；她的第一句话是向玩家讨水。真实历史背景通过角色请求、证词、地图和选择逐步解锁，不在首页或第一轮对话中做百科讲解。

这一入口以移动端为主要游玩表面：手机首屏先呈现角色请求和“回应她”，进入 Space 后先呈现私聊首句与输入区，再下沉角色目录、任务详情和历史场景信息。桌面可以并排扩展辅助信息，但不得要求手机用户先穿过桌面式封面或侧栏才能开始对话。

安妮不是历史人物，历史 Space 也不取代前三座原创世界。前三座继续构成默认发现页的稳定三 Space / 六角色契约；历史试验只是首页当前的聚焦入口。两者共享同一 Space Schema、真实坐标、访问边界和连续性机制，不复制前端假角色数据。

旧的九个功能型公共样板不作为新内容的兼容包袱。已有安装中的旧记录只做可逆归档：保持 `system_public_welfare` 所有权，将 `access` 改为 `private`、`status` 改为 `closed`，从公开发现移除但不删除既有聊天、记忆和状态。新的故事世界使用新 Space / Character ID，避免旧对话被错误映射到新角色。

平台可以版本化更新自己拥有的系统故事，但不能用同一迁移逻辑覆盖用户拥有的 Space、角色卡或世界正史。`owner_id` 因数据所有权和兼容性继续存在，不代表店主供给侧仍是当前产品主线。

## 探索者体验

探索者进入空间前，应先知道：

- 自己当前选择的游玩身份与自声明性别。
- 想见的角色是谁，以及该角色属于哪个完整空间。
- 这个空间大概是什么类型。
- 是否公开、需要密码或不可进入。
- 有哪些 NPC 或主持角色。
- 第一眼应该做什么：打招呼、接任务、找线索、开始仪式、继续回访。

进入空间后，体验重点是：

- NPC 不是被动等用户闲聊，而是主动给出第一步。
- NPC 的称呼、态度、机会和边界应结合玩家游玩身份与自身人设自然变化，但不能套用统一反应或性别刻板印象。
- 对话只是互动方式之一，也可以是选择、轻任务、创作输入或回访确认。
- 记忆、状态和关系应服务于下一次回来，而不是一次性消费。

## 供给侧兼容边界

既有店主 / 创作者管理、导入导出和私有 LLM 配置暂不删除，以保护已有数据和自托管兼容；但当前阶段不为其新增产品入口、增长能力或叙事约束。新故事内容应经过评审后以系统种子或明确发布态进入，运行时 AI 不得直接覆盖 Space、NPC、WorldInfo、玩法、访问规则或访客私密信息。

## 玩法边界

允许：

- Space-local 的轻量文字玩法。
- 访客私有的轻量游玩身份，用作 NPC 对话和玩法分支的运行时上下文。
- 已发布的 Space-local 玩法定义。
- 访客自己的玩法会话。
- AI Director 或 fallback 推进。

不允许：

- 战斗、等级、装备、排行榜、下注、竞技匹配。
- 引入任意脚本执行或不可控游戏中心。
- 把玩法会话变成公开社交状态。

玩法记录应可落库、可回放、可测试。

## 关系与社交边界

FableSpace 关注探索者与空间 / NPC 的关系。

允许：

- 访客私有记忆、回访状态、关系强度、状态卡。
- 既有 owner / system 范围内的空间 / 角色关系边，仅作为兼容数据能力。

不做：

- 好友、私信、动态墙、陌生人匹配。
- 跨空间访客社交图谱。
- 全局在线状态。

## 安全与隐私

- 既有 owner LLM API Key 仍按敏感兼容数据处理，不能返回给访客。
- 平台不做 Token 充值、结算或抽成。
- 私密空间和 Home 信息不能进入公开发现列表。
- 对话、记忆、状态卡和玩法会话要按空间与访客边界隔离。
- 游玩身份和自声明性别不得进入公开 Space payload，不得用作访客匹配、公开筛选或账号权限判断。
- AI 不应生成医疗、法律、金融结论或要求现实危险行动。
- 以儿童作为历史见证者时，必须明确其原创身份，不影射或收集真实未成年人信息；禁止恋爱、暧昧、成人、性化、诱导依附、强迫与血腥猎奇内容。玩家拒绝帮助后仍须有安全的继续或离开路径。

## 验收标准

一个新增能力只有满足以下条件，才算符合空间平台主线：

- 强化游玩身份 -> 角色发现 -> 角色所属完整空间 -> 身份感知互动 -> 记忆 -> 回访链路，同时保留真实坐标锚点。
- 不破坏已发布正史、用户数据所有权和访问边界。
- 不违反 [WHAT_NOT_TO_BUILD.md](WHAT_NOT_TO_BUILD.md)。
- 有明确 Schema、存储边界和 API 权限。
- 可验证：至少有真实命令、测试、截图或人工验收依据。
