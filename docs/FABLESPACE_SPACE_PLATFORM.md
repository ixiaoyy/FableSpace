# FableSpace 角色故事平台主线

本文档定义 FableSpace 当前 P0 的产品主线和能力边界。产品定位见 [PRODUCT_BRIEF.md](PRODUCT_BRIEF.md)，数据合同见 [WORLD_SCHEMA.md](WORLD_SCHEMA.md)，负面边界见 [WHAT_NOT_TO_BUILD.md](WHAT_NOT_TO_BUILD.md)。

## 主线定位

FableSpace 是一个**角色故事平台**。玩家因为想见一个角色而进入其所属的完整故事世界，通过对话和选择建立关系、经历事件，并在回访时延续已有结果。

角色是公开发现入口，故事世界是角色正史、规则、当前事件和连续性的边界。平台不要求玩家先浏览世界目录，也不提供全局身份选择；玩家选择角色后，所属 StoryWorld 的固定 PlayerRole 自动生效。

当前只发布由系统策划、人工审核并可版本化回滚的故事内容。AI 负责在已发布边界内理解自由输入和演绎角色，不负责创建世界正史、永久结局或未审核的剧情状态。

## P0 玩家闭环

```text
首页直接发现角色
  -> 选择想见的角色
  -> 查看一句故事处境与该故事固定身份
  -> 进入角色开场
  -> 自由对话 / 人工编写的关键选择
  -> 写回故事进度、角色关系和私有记忆
  -> 回访时恢复已有结果
```

玩家可以未登录直接开始。匿名进度绑定当前设备认可的访客标识；登录后才承诺绑定账号并支持跨设备恢复。注册或登录不得成为查看角色和开始首轮互动的前置步骤。

## P0 系统故事

当前 P0 只包含两个故事世界：

| StoryWorld | 公开角色 | 固定 PlayerRole | 验证目标 |
|---|---|---|---|
| 1854 年宽街 | 安妮 | 乞丐 | 在不可改写的真实历史时间线内提供原创角色选择和私有后果 |
| 长明宫·雪夜封宫 | 魏观海、萧明珠 | 小太监 | 在原创架空正史内验证多角色立场、关系变化和分支后果 |

安妮是约十岁的原创角色，不是真实历史人物，也不影射真实未成年人。她以讨水这一具体请求带玩家进入宽街事件；史实通过地点、物件、证词、对话和选择逐步呈现，而不是在入场前作为说明材料倾倒给玩家。

雪夜封宫保留“皇帝昏迷、寝殿封锁、未宣诏书”的中央事件。魏观海与萧明珠从不同动机拉拢、试探或拒绝玩家；玩家作为“小太监”可以传话、查证、站队或拒绝，但没有替任何一方开门、宣诏或直接决定正史的权限。

青槐驿与临川大学保留为待重新设计的候选内容。在各自固定 PlayerRole、角色反应、剧情骨架和连续性完成评审前，不进入 P0 公开发现。

## 核心产品对象

| 对象 | 产品职责 |
|---|---|
| StoryWorld | 承载稳定正史、规则、当前事件、角色、固定玩家身份、章节、选择和结局。 |
| Character | 以独立欲望、秘密、语言和处境推动所属 StoryWorld 的事件。 |
| PlayerRole | 定义玩家在一个 StoryWorld 内的固定称谓、性别、背景和入场理由。 |
| PlayerStoryState | 隔离并保存一个玩家在一个 StoryWorld 内的进度、关系、关键选择、私有记忆和回访结果。 |

`NPC` 只描述 Character 由 AI 演绎的运行方式，不是持久化领域实体名。`StoryWorld`、`Character`、`PlayerRole` 和 `PlayerStoryState` 的字段、唯一性和存储约束由 [WORLD_SCHEMA.md](WORLD_SCHEMA.md) 定义。

## 系统内容与发布边界

StoryWorld 是系统内容，不是用户创建的店铺、房间或角色卡容器。当前内容流程是：

```text
人工策划故事合同
  -> 编写并审核正史、角色、PlayerRole、章节、选择、关系规则和结局
  -> 结构校验
  -> 版本化发布
  -> 运行时只读加载
```

系统内容只使用 `draft`、`published`、`archived` 发布生命周期：

- `draft` 不进入公开发现或玩家运行时。
- `published` 可以被新玩家发现并开始。
- `archived` 停止新玩家进入，但保留已有进度、内容版本和回滚依据。

普通用户不能创建、编辑或发布 StoryWorld。当前产品不保留店主、创作者、owner 管理、故事密码、营业状态、内容市场或角色卡导入导出作为隐藏兼容能力。

## 人工剧情骨架与 AI 演绎

故事采用“人工剧情骨架 + AI 角色演绎”的混合推进：

- 系统预先定义章节、节点、关键选择、允许的状态变化、关系规则和结局。
- 玩家既可以点击人工编写的选择，也可以自由输入。
- 确定性规则负责校验剧情动作、前置条件和持久化变化。
- AI 根据 Character、PlayerRole、当前节点、已确认正史和玩家私有连续性生成角色回应。
- 只有已定义且校验通过的剧情动作可以推进章节、改变关键标记或完成结局。

自由输入没有命中剧情动作时仍可形成普通对话和受限的关系反馈，但不能绕过前置条件、创造新正史、跳到未开放节点或直接写入永久结局。任何需要长期保留的状态都必须结构化、可追踪、可回放并可验证。

## 角色关系与连续性

角色关系采用“内部连续好感度 + 故事状态 / 关键标记”的双层模型：

- 自然对话可以依据角色人设和当前语境产生有理由的小幅变化。
- 人工审核的关键选择可以产生较大变化，并打开或关闭分支。
- 重复闲聊不得无成本刷出重大关系阶段或结局。
- 前台不展示好感数值，只展示角色可感知的态度、关系阶段和变化原因。

PlayerStoryState 按玩家与 StoryWorld 隔离，角色关系保存在所属世界内，不跨世界传播，也不进入公开资料、排行榜或访客社交。

故事进行中不能撤销关键选择。到达结局后可以明确开始新一轮；新轮次不继承上一轮的好感度和分支标记，上一轮只在玩家私有状态中保留结局摘要与必要回访记录。P0 不提供存档槽、章节回退或并行时间线。

## 玩家身份边界

每个 P0 StoryWorld 只有一个固定 PlayerRole。它属于故事内容，不是账号权限、现实身份、公开资料或可跨世界复用的职业。

- 进入安妮故事时自动使用“乞丐”。
- 进入魏观海或萧明珠时自动使用“小太监”。
- PlayerRole 的能力必须来自所属世界的时代、制度、背景和入场理由。
- 前台展示简短、可识别的身份名；职责、已知信息和剧情钩子由运行时合同提供。
- 平台不提供全局身份目录、性别选择、用户自定义身份 Prompt 或身份等级系统。

## 历史与原创正史边界

所有 StoryWorld 都有已发布正史，但历史题材与原创架空题材的事实约束不同。

### 历史故事

历史故事运行在经过人工核验的真实时间线上：

| 内容层 | 运行规则 |
|---|---|
| 固定史实 | 经核验的时间或时间范围、地点、真实参与者、可证实的同场关系、制度阶段和已知公开结果；玩家、Character 与 AI 均不得改变。 |
| 可模拟空白 | 明确标记的原创角色、私人对话和不影响已知历史的局部经历；可以产生关系、选择、私有记忆和原创角色后果。 |
| 待核验 | 来源不足、存在实质争议或只有 AI 概括的内容；不得作为剧情因果、角色定论或发布正史。 |

史料没有记载不等于自动获得虚构许可。原创设定发布前必须检查其是否会暗中改变真实人物行踪、事件因果、时代知识、制度阶段或后来已知结果。当前历史故事不提供架空历史分支。

### 原创与架空故事

原创或架空 StoryWorld 不受真实历史时间线约束，但仍受自身已发布正史、角色动机、世界规则、当前事件和允许结局约束。运行时 AI 和玩家声明都不能覆盖这些边界。

## 产品表面

前端只围绕角色发现、角色入场、故事互动、回访恢复和必要的账号绑定展开：

- 首页直接展示安妮、魏观海和萧明珠的真实角色数据，不使用虚构占位角色。
- 角色卡同时给出所属 StoryWorld 和一句当前处境，但不要求玩家先浏览世界目录。
- 进入后优先展示固定 PlayerRole、角色首句、当前选择和输入区。
- 移动端是主要游玩表面；桌面端可以提高密度，不能增加必经步骤。
- 加载、空、失败和回访状态必须真实，不得用示例内容或伪造指标掩盖。
- 产品页面不使用大段说明文案解释控件、系统边界或兜底策略；历史来源和内容分类只在玩家主动打开的参考表面呈现。

## 删除后的能力边界

以下能力不属于当前角色故事平台，也不保留新旧双轨：

- `Space` / `SpaceCharacter` / `VisitorState` 领域合同和 `/spaces` 路由。
- 真实坐标强制字段、地图、LBS、附近发现、路线规划和传统 POI 能力。
- 用户创建、店主 / owner 管理、UGC 发布、内容市场和故事密码。
- SillyTavern 角色卡导入、导出和兼容目标。
- owner 或故事世界级私有 LLM 配置、API Key 管理和店主 Token 统计。
- Home、公开关系、访客好友、私信、动态墙、陌生人匹配和全局在线状态。
- 战斗、等级、装备、竞技排行榜、下注和匹配竞技。

旧代码、路由、配置和开发数据按独立清退任务进行精确审计和删除。开发期不为旧产品协议保留兼容适配器；实际数据清退仍必须有明确目标、备份和可追踪迁移，不能在应用启动时静默删除。

## 安全与隐私

- PlayerStoryState、对话、关系、选择和记忆按玩家与 StoryWorld 隔离，不进入公开发现响应。
- 登录账号和匿名访客标识由服务端可信边界解析，客户端不能注入任意 player ID。
- 系统 LLM 模型、服务地址、API Key 和生成参数由部署环境统一管理；密钥不得进入前端、日志或公开响应。
- AI 输出必须经过剧情动作和持久化边界校验，不能直接修改系统内容。
- 历史儿童角色必须明确为原创，不影射或收集真实未成年人信息；禁止恋爱、暧昧、成人、性化、诱导依附、强迫和血腥猎奇内容。

## 验收标准

新增能力只有同时满足以下条件，才符合当前平台主线：

- 强化“角色发现 -> 固定 PlayerRole -> 对话 / 选择 -> 关系与进度写回 -> 回访”闭环。
- 使用 StoryWorld、Character、PlayerRole 和 PlayerStoryState 统一合同，不恢复旧 Space 双轨。
- 只让人工审核的剧情动作改变关键状态，并让 AI 保持在已发布正史内。
- 保持玩家私有数据按玩家与 StoryWorld 隔离。
- 历史故事不改变经核验的时间线、地点、真实人物关系、制度阶段和公开结果；原创故事不突破自身已发布正史。
- 不违反 [WHAT_NOT_TO_BUILD.md](WHAT_NOT_TO_BUILD.md)。
- 有明确的结构化状态、权限边界和与风险相匹配的真实验证。
