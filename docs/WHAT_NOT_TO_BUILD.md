# FableMap 明确不做清单

本文档是方向约束。新需求如果碰到这些边界，默认不做；确需突破时必须先更新产品文档和 Schema，再实现。

## 核心判断

FableMap 的价值来自：

- 主人主权 > 平台控制
- Token 自主 > 平台计费
- 真实锚点 > 无限自由生成
- 空间 / NPC 关系 > 通用社交网络
- SillyTavern 兼容和可导出 > 平台锁定

## 不做 1：绕过店主确认发布内容

不做：

- 平台自动生成并上线 NPC、空间描述、世界书、剧情或玩法。
- AI 未经确认覆盖、改写或扩写既有空间内容。
- AI 替店主承担最终创作责任。

允许：

- 店主手动配置内容。
- 导入 SillyTavern 角色卡。
- 生成未发布、可编辑、可丢弃、需店主确认的 AI 草稿。

硬规则：店主确认前，AI 草稿不得进入公开 Tavern payload，不得覆盖已有 NPC，不得随空间包导出。

## 不做 2：平台级 Token 付费系统

不做：

- 平台内充值、购买、结算、抽成。
- 平台级 Token 市场。

允许：

- 店主自行配置 LLM 和 API Key。
- 平台记录 owner 可见的 token 使用量。

## 不做 3：无真实坐标的自由空间

不做：

- 脱离真实地图的自由幻想空间。
- AI 创造坐标或漂浮世界。

允许：

- 空间挂接真实 `lat/lon` 后进行主题化、氛围化表达。
- 公开展示时按隐私需求做模糊或抽象表达，但底层仍必须有真实锚点。

## 不做 4：无边界访客社交

不做：

- 好友、私信、动态墙、陌生人匹配。
- 跨空间访客消息。
- 全局在线状态或公开访客社交图谱。
- 绕过店主审批的真人扮演 NPC。

允许：

- 店主审批的单空间 NPC 认领 / 混合扮演。
- 与空间或 NPC 相关的访客私有状态、记忆和回访关系。

## 不做 5：传统地图 App 功能

不做：

- 路线规划、实时导航、POI 评分、商家入驻。
- 为了“地图更炫”而持续堆自绘地图资源和渲染细节。

允许：

- 用地图承载真实坐标、附近空间入口和现实氛围来源。

## 不做 6：传统游戏化系统

不做：

- 战斗系统、等级装备、竞技排行榜、阵营战、下注或匹配竞技。

允许：

- 店主确认的轻量文字玩法、委托、线索、选择式互动。
- Tavern-local 的状态、记忆、任务记录和状态卡。

## 不做 7：不可落库的端到端 AI 幻觉

不做：

- 直接让 LLM 输出最终 UI、Schema 或正史。
- 跳过结构化中间层，产生无法缓存、无法回放、无法测试的结果。

允许：

- AI 生成候选内容、候选状态卡、预览报告或 dry-run。
- 经过确认后写入明确 Schema。

## 快速判断

新需求方向正确，通常会强化：

- 真实坐标上的空间发现。
- 入场、NPC 引导、对话、玩法、记忆、回访。
- 店主创作和确认能力。
- 数据持久化、可导出和可测试。

新需求需要谨慎，通常表现为：

- 主要目标是做社交、导航、战斗、平台计费或地图炫技。
- 让平台替店主发布内容。
- 引入无法落库、无法复核、无法回放的 AI 输出。
