# FableSpace 明确不做清单

本文档是角色故事平台的负面边界。新需求触碰以下方向时默认不做；确需改变产品合同，必须先更新产品简报、平台主线和数据合同，再进入实现。

## 核心判断

FableSpace 的价值排序是：

- 想见一个具体角色 > 浏览功能或世界目录
- 已审核的故事正史与后果 > 运行时无限生成
- 角色差异与关系连续性 > 可替换的统一聊天模板
- 人工剧情骨架 + AI 演绎 > AI 自由决定永久状态
- 玩家私有回访 > 访客公开社交

## 不做 1：运行时 AI 改写正史或永久状态

不做：

- 把玩家声明、角色谎言、模型推测或临场发挥写成 StoryWorld 正史。
- 让 AI 创建新章节、关键选择、永久结局或未审核的状态转换。
- 让自由输入绕过前置条件、跳过剧情节点或直接确认关键标记。
- 用统一 Prompt 抹平 Character 的欲望、秘密、语言、立场和代价。

允许：

- AI 在已发布 Character、PlayerRole、当前节点、正史和私有连续性范围内生成角色回应。
- AI 将自由输入解析为普通对话或候选剧情动作，再由确定性规则校验。
- 经过人工评审、结构校验、版本化并可回滚的系统故事更新。

硬规则：只有已定义且校验通过的剧情动作可以推进章节、改变关键状态或完成结局。

## 不做 2：用户创作、店主与内容市场

不做：

- 用户创建、编辑、导入或发布 StoryWorld。
- 店主 / owner 后台、创建向导、营业状态、故事密码和内容审核队列。
- UGC 内容市场、创作者增长、分成和用户故事包。
- 把历史遗留管理页面作为隐藏兼容能力保留。

允许：

- 系统策划、人工审核并随代码或受控内容注册表版本化发布的故事。
- 内部研究、写作和审校产生的未发布草稿；草稿不得进入公开发现或玩家运行时。

未来若重新引入创作者生态，必须在角色故事核心体验稳定后另行设计权限、审核、版本和回滚合同。

## 不做 3：SillyTavern 与通用角色卡生态

不做：

- SillyTavern 角色卡导入、导出和兼容字段。
- 以通用角色卡格式限制 FableSpace 的 Character 或 StoryWorld 模型。
- 让角色脱离所属 StoryWorld 成为无正史、无事件后果的聊天壳。

允许：

- 为 FableSpace 系统故事保存角色演绎实际需要的名称、人设、动机、秘密、语言、开场和示例对白。
- 对系统内容执行项目自身的结构校验、版本化和回滚。

## 不做 4：地图、坐标与 LBS 产品

不做：

- 为 StoryWorld 强制绑定 `lat` / `lon` 或现实地址。
- 地图首页、附近发现、坐标创建、路线规划、实时导航和 POI 搜索。
- 为了地图展示持续堆叠自绘资源、定位权限或地理服务依赖。
- 把历史地点记录重新解释为通用坐标 Schema 或 LBS 入口。

允许：

- 历史故事把经核验的真实地点作为内容事实并附来源。
- 原创或架空 StoryWorld 使用自己的场景与世界规则，不需要现实地理锚点。

## 不做 5：全局或用户自定义玩家身份

不做：

- 全平台身份目录、通用性别选择和跨 StoryWorld 复用的职业身份。
- 用户提交任意身份 Prompt、背景或能力声明。
- 把“乞丐”“小太监”等 PlayerRole 变成账号权限、现实身份、公开资料或等级职业。
- 让 PlayerRole 绕过剧情条件、时代制度或角色权力边界。

允许：

- 每个 StoryWorld 提供一个经过审核的固定 PlayerRole。
- PlayerRole 以所属故事的称谓、性别、背景、入场理由和可见信息影响角色回应。

## 不做 6：旧 Space 合同与兼容双轨

不做：

- 新增或继续维护 `Space`、`SpaceCharacter`、`VisitorState`、`space_id` 和 `/spaces` 产品合同。
- 为旧中文路径、旧字段、旧 API 或开发期数据保留永久别名和兼容适配器。
- 在新代码中同时使用 StoryWorld 与 Space 表达同一领域概念。
- 让旧 owner、访问密码、营业状态或地图字段渗入新 StoryWorld Schema。

允许：

- 在独立清退任务中对旧入口、引用、表、配置和部署项做只读审计。
- 通过明确迁移和备份清退旧开发数据；不得在应用启动时静默删除。

## 不做 7：无边界访客社交

不做：

- 好友、私信、动态墙、陌生人匹配和跨故事访客消息。
- 全局在线状态、公开关系图谱、公开好感榜或玩家排行榜。
- 将 PlayerStoryState、对话、选择、记忆或角色关系暴露给其他玩家。

允许：

- 玩家与 Character 之间按 StoryWorld 隔离的私有关系、记忆、选择和回访。
- 登录后将自己的匿名进度绑定账号并跨设备恢复。

## 不做 8：平台付费与私有 LLM 配置产品

不做：

- Token 充值、购买、结算、抽成和模型调用市场。
- owner 或 StoryWorld 级 API Key、模型、服务地址和生成参数配置。
- 面向店主的 Token 用量或计费面板。

允许：

- 由部署环境统一管理系统 LLM 配置和密钥。
- 不泄露密钥、玩家内容或隐藏上下文的必要运维日志与指标。

## 不做 9：不可追踪、不可回放的 AI 状态

不做：

- 让 LLM 直接输出并持久化最终 Schema、正史或任意状态对象。
- 跳过结构化剧情动作和校验层写入章节、关系、记忆或结局。
- 保存无法解释来源、无法回放或无法验证的永久结果。

允许：

- AI 生成角色可观察对白和受约束的候选解释。
- 确定性规则记录剧情动作、状态变化、原因和来源。
- 经过筛选并绑定当前玩家、StoryWorld、StoryRun 与来源的私有记忆。

## 不做 10：可刷分的关系与传统游戏化

不做：

- 向玩家展示可刷取的好感数值。
- 通过重复闲聊无成本获得重大关系阶段、关键分支或结局。
- 战斗、等级、装备、竞技排行榜、阵营战、下注和匹配竞技。
- 存档槽、章节回退或并行时间线。

允许：

- 自然对话产生有依据的小幅关系变化。
- 人工审核的关键选择显著改变关系并打开或关闭分支。
- 前台展示角色态度、关系阶段和变化原因。
- 结局后明确开始新一轮，并仅保留上一轮结局摘要。

## 不做 11：历史故事的架空改写

不做：

- 移动、合并、提前或延后真实事件的时间、地点、参与者、制度阶段与已知结果。
- 让玩家阻止、促成、取代或亲自造成已有明确结果的真实历史事件。
- 让没有同场证据的真实人物见面，或把主题相似写成真实交流、影响与因果。
- 把史料空白、史学争议、AI 推测、原创对白或人物动机伪装成历史事实。
- 让跨时代知识直接改变当时的技术、制度或公开历史结果。

允许：

- 在经核验的真实时间线内使用明确标记的原创角色、私人对话和不与史料冲突的局部事件。
- 改变原创角色的信任、关系、选择、是否得到帮助和私有回访结果。
- 将来源不足或存在实质争议的内容保留为 `待核验`，核验前不进入剧情因果或角色定论。

硬规则：当前历史故事不提供架空历史分支。任何分支只要会改变固定史实，就不得发布为历史内容。

## 不做 12：用假内容掩盖产品状态

不做：

- 用虚构角色、占位 StoryWorld、伪造统计、伪时间或示例记忆填充首页。
- 在加载、空或错误状态中渲染看似可进入的故事卡。
- 用大段说明文案代替清晰的信息层级和真实可操作状态。
- 向公开响应或界面暴露其他玩家的进度、对话、选择和记忆。

允许：

- 展示来自已发布系统内容的真实角色、StoryWorld 和入场信息。
- 使用明确、简洁、不可误认为内容实体的加载、空、错误和重试状态。
- 在玩家主动打开的历史参考表面展示来源及“史实 / 剧情设定 / 待核验”分类。

## 快速判断

方向正确的需求通常会强化：

- 角色发现、固定 PlayerRole、开场对话和关键选择。
- Character 的独立立场、语言、关系变化与事件后果。
- PlayerStoryState 的私有进度、记忆、回访和账号绑定。
- 人工剧情动作、结构化持久化、回放和验证。
- 历史固定正史或原创世界已发布正史的稳定边界。

需要停止并回到产品合同审查的需求通常表现为：

- 主要目标是地图、UGC、店主管理、角色卡生态、社交、平台计费或传统游戏化。
- 恢复 Space 兼容层或让新旧领域协议长期并存。
- 让玩家输入或运行时 AI 未经审核改变正史、关键状态或永久结局。
- 引入无法隔离、无法追踪、无法回放或无法验证的数据流。
